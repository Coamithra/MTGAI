# MTGAI Project Conventions

> **ASD is legacy — do not use it.** ASD was the original hand-built set and the project's first target; the toolchain is now set-agnostic and ASD is not used for anything. It is no longer a default `--set`/`set_code` value anywhere in the app (the CLI resolves the set code from the active project; artifact paths come from the project's `asset_folder`). It survives only as the pre-existing tracked files under `output/sets/ASD/` (`cards/`, `mechanics/`, `reports/`) — kept purely as a historical artefact — plus a few incidental mentions in standalone dev scripts and historical docs/learnings. Treat all of it as dead weight: don't build on it, don't generate into it, and don't cite it as the example set. New work uses the wizard's per-project `.mtg` flow with the user-picked asset folder.

## Project Structure
- Backend: `backend/mtgai/`. Tests: `backend/tests/` (mirrors source). Research: `research/`. Learnings: `learnings/`. Plans: `plans/`. Task tracking lives on Trello (see above) — `plans/TRACKER.md` is deprecated (history in git).
- All `output/` is gitignored. Pre-existing tracked files (ASD's `cards/`, `mechanics/`, `reports/`) stay tracked; new artifacts in `output/` won't be picked up.

## Development
- Python 3.12+ (uv-managed). Lint: `ruff check .` / `ruff format .` from `backend/`. Tests: `pytest` from `backend/`.

## Code Style
- Pydantic v2 for all data structures. StrEnum for enumerations.
- `X | Y` union syntax (not `Union[X, Y]`). `list[X]` (not `List[X]`).
- Line length 100. Imports sorted by Ruff (isort).

## Data Models
- Card schema is the single source of truth: `mtgai/models/card.py`. Field names match Scryfall (`oracle_text`, not `rules_text`).
- Card data: JSON files in `output/sets/<SET_CODE>/cards/`.
- Naming: `<collector_number>_<card_name_slug>.json`; art: `..._v<attempt>.png`; render: `....png`.
- **Repo-relative paths**: never hardcode an absolute path like `Path("C:/Programming/MTGAI/...")` — it breaks on any other checkout / CI. `mtgai/io/paths.py` is the canonical seam: `repo_root()` (the repo root, via `Path(__file__).resolve().parents[3]` from `mtgai/io/`) and `output_root()` (`repo_root()/"output"`, the gitignored sibling of `backend/`). Library code imports these; standalone `backend/scripts/*.py` (not on the package path) inline `Path(__file__).resolve().parents[2]`. A file's own bundled assets (prompt dirs, art workflows) use `Path(__file__).resolve().parent / ...`. External tool roots stay machine-specific but read from env where trivial (e.g. `COMFYUI_ROOT` env var, default `C:/Programming/ComfyUI`).

## Architecture overview

The codebase is a multi-stage pipeline that turns a setting prose document into a finished MTG set (cards + art + rendered images). Stages are orchestrated by `mtgai/pipeline/` and surfaced at `/pipeline` in the local FastAPI server (`python -m mtgai.review serve --open`). Each stage is independently runnable from the CLI. The toolchain is set-agnostic; the project is driven by the wizard's per-project `.mtg` flow.

Card statuses: `draft → validated → approved → art_generated → rendered → print_ready`. Pipeline is resumable; every generation attempt is tracked.

**Re-entrant pipeline / review→regen loop**: a stage can appear more than once.
- The post-`card_gen` review **gates** (the merged Conformance & Interactions gate — two internal LLM steps — then Design Review) check card-vs-slot conformance, whole-pool interactions, and design quality; on failure they stamp `regen_reason`/`flagged_by` onto the offending cards and set `StageResult.rerun_from="card_gen"`.
- The engine then *appends a fresh instance span* `[card_gen … that gate]` right after the flagging step and walks forward into it (forward-only, never a backward jump).
- `StageState.instance_id` is the per-instance/tab id — `== stage_id` for the **backbone**, `f"{stage_id}.{n}"` (e.g. `conformance.2`, displayed "Conformance & Interactions 2") for inserted copies; the shared `stage_id` still keys the runner, clearer, model assignment, and break-point. Each instance is its own wizard tab.
- A gate that hits `MAX_REVIEW_ROUNDS` (engine.py) with cards still flagged pauses *that instance* for human review (best attempt kept, cards left flagged).
- `card_gen` treats a slot whose card has `regen_reason` as needing regeneration: it archives the prior card, threads the reason into the prompt, and clears the flag on the new card.
- **Per-instance card-pool version tracking** (`pipeline/history.py`): the live `<asset>/cards/` folder stays the single working set (always the loop tip), but on completion each loop instance (`lands` + `card_gen` + the `conformance` & `ai_review` gates = `SNAPSHOT_STAGES`) snapshots the live `cards/*.json` + `generation_progress.json` to a write-once `<asset>/history/<instance_id>/` at the engine `save_state` seam (`_snapshot_instance_output`; `_regen_archive/` excluded; plain copies). An instance's *entry* pool is its predecessor's snapshot, pinned as `StageState.entry_snapshot_id` when it starts. `engine.rerun_instance(state, id)` re-runs any duplicable instance (`RERUNNABLE_STAGES` = the three loop stages) from that entry: restore the snapshot → truncate downstream (+ their history) → reset the instance → re-append the canonical forward path via `_build_forward_path` (the *forward* mirror of `_build_rerun_span`, auto-numbered so a surviving sibling yields `conformance.2` while a fresh stage yields backbone `ai_review`; loop stages stay AUTO, post-loop stages honour break points) → set `NOT_STARTED` so the caller's `_kickoff_pipeline_engine` walks forward from it. Surfaced by `POST /api/wizard/instance/rerun {instance_id}` (409 if engine/extraction running; same cascade contract as `edit/accept`) + the per-tab "Re-run this step" button (`W.rerunButtonHtml`/`bindRerunButton`/`rerunInstance` in `wizard_util.js`). Tab card-views read `history/<id>/` for a completed *non-tip* instance and live `cards/` for the tip (`?instance_id=` on `card_gen/state` → `_resolve_view_cards_dir`), fixing the stale "old tab shows the latest cards" view. Pre-version-tracking projects (no `entry_snapshot_id`/`history/`) disable the button and degrade to a from-live re-run. The Card Generation tab uses this same machinery for its progress + old/new view of a regen instance: it scopes the "X / Y cards generated" headline to the slots THAT instance (re)creates (`generate_set` reports `run_target = len(unfilled)`, not the full set, so a regen instance reads "5 / 5" not "5 / 277"), and highlights the cards an instance regenerated vs carried over by diffing its pool against its `entry_snapshot_id` (`/api/wizard/card_gen/state` returns `is_regen_instance` + a per-card `is_new`; the per-card SSE stream marks streamed cards `is_new=True`; the highlight is suppressed on the first `card_gen` / pre-version-tracking projects, where "everything is new" is not a useful distinction).
- The generic edit-cascade (`_apply_cascade_clear`) clears card_gen via the **scoped** `clear_card_gen` → `clear_card_gen_cards()` (card_gen-owned `cards/*.json` only, plus `generation_progress.json` + `_regen_archive/`, **preserving the lands stage's `L-*`** — shared with the `card_gen/refresh` endpoint via `stages._is_land_stage_card`), and drops the cleared instances' `history/` snapshots; the rerun path above is the precise per-instance reset that restores entry snapshots instead of deleting.

**Key modules** (look in the file for detail; CLI flags via `--help`):

- `mtgai/pipeline/` — end-to-end orchestrator: stage registry, FastAPI server, SSE event bus, wizard shell. State persisted to `<asset_folder>/pipeline-state.json` (the asset folder the user picked on Project Settings).
- `mtgai/pipeline/wizard.py` — wizard state resolver. Maps the on-disk `pipeline-state.json` + `theme.json` into the visible-tabs / latest-tab shape `wizard.html` consumes. URL routing lives in `pipeline/server.py` (`/pipeline/<tab_id>`). The legacy `/pipeline/configure` route is a theme-aware redirect into the wizard.
  - Project Settings (kickoff) tab payload comes from `GET /api/wizard/project`; live-apply edits go through `/api/wizard/project/{params,theme-input,breaks,models,preset/apply,preset/save}`; `POST /api/wizard/project/start` validates + spawns the theme extractor and tells the client to navigate to `/pipeline/theme`. The extraction worker, when launched via the kickoff path, also writes the assembled theme.json on `done` so the Theme tab finds populated content on next mount.
  - **Building or modifying a wizard tab? Read `plans/wizard-tab-conventions.md` first** — it documents the patterns Project Settings + Theme establish (single Save & Continue footer button, required-field asterisks, form lock during AI gen, AI provenance badges, past-tab edit cascade, streamed-LLM textareas, etc.) so new tabs stay consistent.
- `mtgai/pipeline/theme_extractor.py` — theme content extraction. Surfaced as the Theme tab inside the wizard. Uploads PDF/text, runs multi-stage LLM extraction, then JSON pass for constraints + card suggestions.
- `mtgai/generation/llm_client.py` — all LLM transport routes through [llmfacade](https://github.com/Coamithra/LLMFacade) (Anthropic + llama.cpp). Single entry point `generate_with_tool()`; provider resolved per-call from the model registry.
  - **Prompt caching primitive**: pass keyword-only `system_blocks=[(text, cache), ...]` (each item a plain `str` = uncached, or a `(text, cache)` tuple) instead of `system_prompt` to place bulk STATIC context in its own independently-cached Anthropic system block, and/or `cache_user=True` to cache the last user block. Anthropic caps a request at **4 `cache_control` markers** (tools + system blocks + optional last user), so bundle all static context into **ONE** block. Both are no-ops on llamacpp (blocks flattened to one joined system string, cache flags dropped, `cache_*` stay 0). `card_gen` uses it via `prompts.build_static_set_context()` — the ~6-7k-token set-context prefix (setting prose + mechanics + archetypes + preventive guidance) rides in a cached system block hoisted out of the batch loop, written once and read at ~0.1x across the ~200-250 per-set batches; the dynamic per-batch content (existing-card context, cycle siblings, slot specs) stays in the user message via `build_user_prompt()`. The mechanics council's same-round reviewers use `cache_user=True` (byte-identical user prompt). The structural stages follow the same split (static set context in one cached system block, dynamic trigger/data in the user turn): `skeleton_knobs_tuner` (context = `skeleton_knobs_context.txt` formatters), `reprint_selector` select (context = set framing + the ~350-card pool) and place (context = chosen reprints + slot list), and `land_generator` basics + investigation. `archetypes` and the skeleton `relabel` pass already keep their static context in the system prompt (auto-cached), so they were audit-only. These are low call-volume (1-6 calls/stage), so the win is consistency + retry/re-roll reuse within the TTL, not the per-set cost win.
- `mtgai/settings/` — model registry (`models.toml`) + the active project's `ModelSettings` (in-memory only; the .mtg file is the persistent store) + global default-preset (`output/settings/global.toml`).
  - The settings shape carries `[llm_assignments]` / `[image_assignments]` / `[effort_overrides]` / `[break_points]` / `[set_params]` (set_name + set_size + mechanic_count) / `[theme_input]` (kind + filename + upload_id + char_count + uploaded_at) / `asset_folder`.
  - All resolution goes through `get_active_settings()` (or `require_active_project().settings`); helpers no longer take `set_code`. Live-apply endpoints call `apply_settings(new)` to update the pointer; the user saves the .mtg via the browser's File System Access API to persist.
  - Profiles save with `profile_only=True` so they exclude `set_params` + `theme_input` (per-project, not template-able). Cross-set defaults UI at `/settings`; per-project Project Settings UI is the wizard's first tab (`/pipeline/project`, endpoints under `/api/wizard/project/*`).
- `mtgai/validation/` — two-tier validators (AUTO auto-fixed; MANUAL becomes LLM retry prompts). Cards immutable; fixers return new instances. The `validate_card` runner sequence is schema → mana → type_check → rules_text → keyword_ordering → text_overflow → uniqueness. `keyword_ordering` (`validation/keyword_ordering.py`) is an AUTO check: keyword-ability lines (evergreen + the set custom mechanics, via `rules_text.all_keywords()`) must sit **above** complex triggered/activated/static abilities; the fixer stable-partitions keyword lines to the top, moving each line whole so injected `(reminder)` text rides along untouched (per the validators-skip-parenthesized-text contract). The custom-keyword vocabulary is resolved per-call from the active project's `mechanics/approved.json` mechanic names (`rules_text.custom_keywords()`, cached by path+mtime; `set_custom_keywords()` overrides it for tests/callers), so the validators recognize **any** set's keywords — not a hardcoded seed. The card-gen system prompt (`research/prompt-templates/system-prompt-v1.md`) also instructs the model to template keywords first, but the validator is the durable backstop against prompt drift.
- `mtgai/generation/reminder_injector.py` — reminder text is **never LLM-generated**, always injected programmatically from mechanic definitions after review.
- `mtgai/review/finalize.py` — post-review: inject reminder → validate → auto-fix → save. Writes a JSON sidecar `<asset>/reports/finalize-report.json` (alongside the markdown report) that the **Finalization wizard tab** reads to badge auto-edited cards + show the before/after of what the stage changed (`original_oracle_text` is kept per modified card). The `finalize` stage is `review_eligible: True` so the user can toggle "Stop after this step" to pause and **edit any field of any card by hand** in the tab (`wizard_finalize.js` + `/api/wizard/finalize/{state,save-card,save}`); the break-point defaults off, so finalize still auto-runs unless the user opts in. Manual-edit markers persist to `<asset>/reports/finalize-user-edits.json` so the "edited" badge survives reload.
- `mtgai/review/ai_review.py` — tiered LLM review (single-reviewer for C/U, full council for R/M + planeswalkers/sagas), iteration loop, resumable. In the review loop it's the **hybrid Design Review gate**: the council **revises in place** (its measured strength) as the primary action, and only a card still rated REVISE after `MAX_ITERATIONS` is flagged (`flagged_by="ai_review"`, `rerun_from="card_gen"`) for a from-scratch regen — the loop is the overflow path, not the primary one (`review_all_cards` returns the `unfixable` list the runner flags). **AI Design Review tab** (`wizard_ai_review.js`, stage_id `ai_review`): a per-card review grid mirroring card_gen tiles, with a green-check "approved" / red-X "rejected" stamp per card (rejected = flagged-for-regen or a still-REVISE verdict; the rejection reason shows under the tile) and a muted "to review" state for not-yet-reviewed cards. While the stage runs, each card streams a **live council** (`ai_review_council` thumbs, mirroring the mechanics council via `build_ai_review_hooks` threaded through `review_set`); the per-card submenu offers **Approve** / **Revise...** (an inline textbox -> one LLM `revise_card_in_place` call, mirroring council revise-in-place) / **Regenerate** (`_flag_cards_for_regen` -> the slot bounces to card_gen). User decisions persist in a `reviews/decisions.json` sidecar (`load_decisions`/`save_decision`), merged over the AI verdict by `GET /api/wizard/ai_review/state`; manual actions are `POST /api/wizard/ai_review/{approve,revise,regenerate}`. SSE events: `ai_review_reset` / `ai_review_card_start` / `ai_review_council` / `ai_review_card_done`.
- `mtgai/analysis/` — the **merged Conformance & Interactions gate** (stage_id `conformance`, display "Conformance & Interactions"), run by `stages.run_conformance` as two whole-set LLM steps plus a purely-algorithmic duplicate scan (no LLM) that runs first and folds its hits into the conformance checklist, all under one AI-lock hold + one poller (the two LLM steps resolve the shared `conformance` model assignment):
  - `conformance.py` — each card vs. its slot's `tweaked_text` spec; holistic, **no descriptor parser**; the gate's first step.
  - `interactions.py` — whole-pool degenerate-combo scan naming the enabler card + a `replacement_constraint`; the gate's second step.
  - `duplicates.py` — algorithmic functional-duplicate scan (runs **first**, **no LLM call**, **no section of its own**): builds a per-card signature `(normalized type_line, P/T/loyalty, normalized oracle text)` that **excludes mana cost** ("functionally the same modulo mana cost"), sorts oracle clauses + folds the card's own name so keyword/clause reordering and self-references don't hide a dup, groups, keeps the lowest collector number per group and flags the rest. The hits are passed to `check_conformance(pre_flagged=...)`, which seeds each as a pre-failed (X) row in the per-card Conformance checklist, skips that card's per-card LLM call, and returns it as a conformance finding — so duplicates surface inside Conformance, not a separate step. A normalization, not a semantic compare — a genuinely-reworded dup slips through (accepted trade-off for a free, instant pass). Skips basics + reprints like the other steps.
  - The two LLM steps skip basic lands + reprints (`gate_common.filter_gate_cards`) and go through `gate_common.generate_gate_tool` (retries a truncated / repetition-looped local response, bumping temperature; raises after the budget so the stage fails visibly rather than skipping a check). The runner combines the conformance (duplicates folded in) + interaction findings (a card flagged by both keeps every reason), stamps them via `_flag_cards_for_regen` (`flagged_by="conformance"`), exposes per-step `artifacts["steps"]` for the tab (two steps: conformance + interactions), and bounces to `card_gen` if either flagged a card. The old separate `balance` stage_id is retired.
  - Per-card design-quality heuristics live in `heuristic_checks.py` (consumed by AI review; the old render QA stage that also used them was dropped in the art/render topology reorg). Set-level balance is now an upstream skeleton invariant + conformance — the old set-level balance/coverage/algorithmic-conformance machinery and the `mtgai review balance` CLI are gone.
- **Mechanics** (`mtgai/generation/mechanic_generator.py`, the `mechanics` stage) — generates candidate mechanics, then AI-picks the best N via `pick_best_mechanics` and writes `approved.json` + sidecars itself through `persist_mechanic_selection`, so it is **not a structural break-point** (its pause defaults on but is user-toggleable). The wizard pre-selects the AI picks + shows the rationale; "Re-pick with AI" hits `POST /api/wizard/mechanics/pick`.
  - Each draft is gated by a per-candidate **council** (`council_review`, theme-free by design — ported from `mtg-mech-lab`): `MECHANIC_COUNCIL_SIZE` (3) independent reviewers sanity-check it as a bug-catcher, not a taste critic: a workable, rules-sound mechanic passes even if simple or unoriginal, and a reviewer's REVISE only blocks when it cites a concrete **`major`** defect in a gating category (playable / wording / self_consistent; `elegant` and any `minor`-severity nit are advisory, surfaced but never blocking; the old taste standards `interesting`/`unique` are gone — see `_effective_verdict` + `MECHANIC_BLOCKING_CATEGORIES`). A synthesizer then revises in place (at the `HEAVY` budget) and the council re-reviews, up to `MAX_MECHANIC_REVIEW_ITERATIONS` (3) rounds. The loop exits when **no reviewer raises an open `major` blocking defect** (a lone major from any reviewer blocks — it can't be outvoted, since objective defects like a dead-on-use example must not lose to reviewers who missed them — while `minor` nits never block), never on the synth's self-verdict (which over-claims); on the final round before a from-scratch regen the otherwise-discarded synth is skipped (`skip_final_synth`). A draft still REVISE after the budget is **regenerated from scratch** (the `card_gen` regenerate-with-reason pattern — the council's reasons thread into the gen prompt) up to `MAX_MECHANIC_REGEN_ATTEMPTS` (1) times, then accepted best-effort flagged REVISE so the slot still fills. The pool ends up (near-)all council-passing, which gives `pick_best_mechanics` real choice on **theme fit** (the dimension the theme-blind council can't judge). Every call is best-effort (a bad pass never destroys a good draft); the anti-rename guard holds; `_review_verdict` + `_review_notes` stamp onto the mechanic for the wizard. Synth deviates from the lab in two **deliberate** ways (see `learnings/reasoning-budget-overrun.md`): `max_tokens=HEAVY` (16384, ~2× the synth's real need, not the lab's 24576 — respects the `token_budgets` ceiling + Anthropic's non-streaming threshold), and **no** `repeat_penalty` escalation (verified not to fix that truncation; baseline 1.1 stays). Generation runs at temp 0.9 (beats 1.1), reviewers 0.4, synth 0.2. Always-full-council, so the mechanics stage is the most LLM-call-heavy — assign a capable `mechanics` model if local runtime hurts.
  - Candidates **stream live** into the wizard via three SSE events: `mechanic_candidates_reset` (clears the strip; only fires on a full from-scratch run), `mechanic_candidate_drafted` (pre-review; card appears with a "Reviewing…" badge), `mechanic_candidate_finalized` (post-review; card replaced in-place, optional "Reviewer tweak: …" line if `review_notes` non-empty). Engine path wires the hooks via `StageEmitter.event`; refresh endpoints route them through `event_bus.publish` with the slot-index remapped to the row being refreshed.
- **Archetypes** (`mtgai/generation/archetype_generator.py`, the `archetypes` stage between `mechanics` and `skeleton`) — writes `<asset>/archetypes.json`, one `DraftArchetype`-shaped entry per color pair: `color_pair` + `name` + free-text `description` (the intent — **wincon-first**, may name mechanics it leans on, pace inherent so no `speed` field). Signpost uncommons are a **skeleton** concern, not an archetype field: `generate_skeleton` flags one multicolor-uncommon slot per pair via `SkeletonSlot.signpost_for`, and `prompts.format_slot_specs` turns that flag + the pair's archetype intent into a "design the gold uncommon that defines this archetype" card-gen instruction.
- **Skeleton Generation** (`skeleton/generator.py` + `skeleton/knobs.py` + `generation/skeleton_knobs_tuner.py` + `generation/skeleton_relabel.py`, the `skeleton` stage between `archetypes` and `reprints`) — one stage, **three phases**, all under one AI-lock hold. Model assignment `skeleton` (shared by all phases); transcripts → `<asset>/skeleton/logs`. Shared set-context prompt blocks (setting / mechanics / archetypes / constraints / requests) live in `generation/skeleton_prompt_blocks.py` (also used by the reprint stage). Prompts: `pipeline/prompts/skeleton_{knobs,relabel,assign}_{system,user}.txt`. Design: `plans/skeleton-generation.md` + `plans/skeleton-knobs.md`.
  - **Phase 0 — Knob tuning** (`skeleton_knobs_tuner.tune_skeleton_knobs`): an LLM tunes the structural **knobs** (`SkeletonKnobs`) to the theme via one tool call — rarity weights, multicolor / colorless / creature fractions, non-creature type bias, planeswalker count, signposts-per-pair, and cycles. The LLM proposes; the schema clamps and `generate_skeleton` reconciles to the hard invariants, so an out-of-range value can never produce an illegal skeleton. Defaults reproduce the historical skeleton on the constraint-bearing dimensions (per-color balance, creature density, signpost coverage, rarity totals); `planeswalker_count` defaults to **0**. Failure → default knobs + `SkeletonResult.knobs_defaulted` (a tab notice, never a hard error). The `KNOB_SPECS` registry (`skeleton/knobs.py`) is the single source of bounds driving AI-output validation, manual-input validation, and the wizard control bounds.
  - **Cycles** (`Cycle`/`CycleSpan`) — balance-preserving structural reservations, always a multi-card family (one-per-color `mono5`, one-per-pair `pairs10`, allied/enemy fives, shard/wedge three-color fives — **no single-card span**; a one-off card is just a card, pinned via a theme `card_request`). Carved out of the rarity budget first; stamp each member's `cycle_id` + `cycle_name` + `cycle_member` (the member's colour slot — `W` / `WG` / `WUB`). The name + member surface as a `· cycle:<name> (WG)` tag on `render_slot_string`, so the relabel sees the family *and each member's place in it* the same way it sees `signpost:<pair>`; the shared template is applied later at card-gen. This is how a set states a bold structural gesture (a 10-card guildgate land cycle, a mono-5 rare titan cycle) without breaking balance.
  - **Phase 1 — `generate_skeleton(config, knobs)`**: builds the deterministic, balanced **default** skeleton from the project's `set_params` (sized by `set_size`, *not* theme.json) and the tuned knobs. It also accepts `reserved_slots`: `build_reserved_slots(theme)` turns `theme.json` `card_requests` + structured `legendary_characters`/`notable_cards` anchors into `ReservedSlotSpec`s; matching slots get a `reserved_card` stamp surfaced into the card-gen prompt by `format_slot_specs`, without changing slot counts/color/rarity/type so balance holds.
  - **Phase 2 — `skeleton_relabel.relabel_skeleton()`** runs the LLM **relabel**: each slot's one-line descriptor (`render_slot_string`, e.g. `"White · common · creature · CMC1 · vanilla"`) is rewritten to fit the set. Pass 1 relabels every descriptor (named mechanic instead of a bare tier, color/type/rarity swings, legendaries; count-invariant, reconciled by `slot_id`); Pass 2 places `theme.json` `card_requests` onto slots. Pass 1 **token-streams** via `llm_client.stream_text` (free text, not a tool call): each `--CARD <id>--` block is parsed the moment it closes and pushed to the Skeleton tab as a `skeleton_slot` SSE event (`skeleton_relabel_reset` per attempt, `skeleton_relabel_done` at the end), wired via `StageEmitter.event` / `event_bus.publish` + the `on_slot`/`on_reset` hooks. The rewrite is stored per slot as `SkeletonSlot.tweaked_text`; the structured fields stay the default, so `reprints`/`lands` read them unchanged.
  - **Card-gen consumption**: card generation reads `tweaked_text` as the slot's spec via `prompts.format_slot_specs` (else falls back to `render_slot_string`). `group_slots_into_batches` batches by the (default) structured color **but pulls each cycle's members into their own batch** so the family is designed together with its shared `template` (threaded as a CYCLE MEMBER instruction). Land cycles (members carrying the deterministic `card_type == "land"` seed — e.g. a guildgate `pairs10`) are generated by card-gen like any other cycle.
  - **Failure handling**: the default skeleton is saved before the relabel, so a relabel failure leaves a usable un-themed skeleton. A partial relabel is *kept* — unreached slots fall back to their default descriptor and the run is flagged `SkeletonResult.relabel_incomplete` (persisted on `skeleton.json`, surfaced as a tab warning that survives reload). Only a zero-output run (every attempt errored) is a hard error.

## Cross-cutting

- **AI mutex (`mtgai/runtime/ai_lock.py`)**: app-wide, **one AI call at a time**. All AI-touching endpoints share it. Conflicting requests get 409 with a busy payload. Wizard refresh/regenerate endpoints wrap their AI work in `pipeline/server.py`'s `with guarded_ai("Action name", stage_id=…) as guard: if guard.busy: return guard.busy_response` — the context manager owns acquire→409 + the `try/500 {"error": …}` envelope (worker exceptions become `AIActionError`, rendered by the registered handler) + the success-only `_heal_failed_stage(stage_id)` + release. Suppress the heal after a cancelled run with `guard.skip_heal = True` (the card_gen contract). The engine path (`stages.py`) still uses `ai_lock.hold(...)` directly. The single busy-409 helpers are `_busy_response()` and `_reject_if_busy()` (the latter for gate-only endpoints that don't hold the lock — theme section-refresh / kickoff). Long-running workers (card_gen, mechanics, lands, reprints, skeleton relabel, conformance, ai_review, art_prompts, art_gen) poll `ai_lock.is_cancelled()` at their loop/call boundaries and stop early (keeping partial output) so the Cancel button actually halts them; the guard then skips the heal. **Strictly non-reentrant** (NOT an `RLock`): a thread already holding the lock that calls `hold()`/`try_acquire()` again gets busy back (`None`) and the nested body is silently skipped — acquire once at the outermost layer. `try_acquire()`/`hold()` return a monotonic `run_id` (`int` >= 1 on success, `None` when busy; truthy/`is not None` checks both work); `request_cancel(run_id=...)` only cancels if the active run matches that id, so a stale cancel can't kill a newer run that reused the lock. `request_cancel()` with no arg keeps the legacy "cancel whatever's running now" semantics (used by the UI cancel button + project-switch drain).
- **Tab state (`mtgai/runtime/runtime_state.py`, `extraction_run.py`)**: server is the source of truth for AI run state, pipeline state, and saved theme.json; browser holds UI ephemera in localStorage. SSE streams are broadcastable (tab-switching mid-run does not cancel; cancel is opt-in via the explicit button). New long-running AI runs should follow the same broadcast pattern. Aggregator: `GET /api/runtime/state`.
- **Active project**: which `.mtg` is open is the source of truth. The server holds a process-memory `ProjectState` pointer (`mtgai/runtime/active_project.py` — set_code + settings + .mtg path) — set by `/api/project/{open,materialize}`, cleared by `/api/project/new`, lost on restart so a fresh boot always greets the user with New / Open. Stage outputs land under the project's `asset_folder`; `mtgai/io/asset_paths.set_artifact_dir()` reads the active project and raises `NoAssetFolderError` (translated to 409 at endpoints) if none is open. Switching projects mid-AI-run requires `force=true` in the request body; without it the endpoint returns 409 + the AI lock's busy payload so the client can render a confirm modal. The cancel path calls `ai_lock.request_cancel()` then `active_project.await_lock_release()` before flipping the pointer.
- **Reminder text**: not validated, not LLM-generated. Stripped + injected by `reminder_injector` after review. Validators that touch oracle text must skip parenthesized text.
- **Progress UI**: the wizard's **global progress strip** (`#wiz-progress-strip` in `wizard.js`, sticky at top) is the one canonical progress surface. Driven two ways:
  1. SSE `phase` events on `/api/pipeline/events` → `handlePhaseEvent` — fed by `StageEmitter.phase()` for engine stages and `event_bus.stage_phase()` for tab-refresh. **Live token-rate stats** (prompt-eval / generation tok/s) come from the shared `PromptEvalPoller` (`generation/phase_poller.py`), which polls llama-server's `/slots` on a daemon thread and emits `generation`/`prompt_eval` ticks. **Every local-model stage is wrapped**, both paths: engine runners (`stages.py`) via `make_poller(stage, emitter.phase, —)`, and tab-refresh endpoints (`server.py`) via the `_bus_poller(stage_id, —)` context manager (routes the poller through `event_bus` + emits a terminal `phase:"done"` so a trailing tick can't leave the strip stuck). `make_poller` resolves the stage's assigned model and no-ops (`NullPoller`) for a cloud model / no open project — so Anthropic stages just get the indeterminate bar. `theme_extractor` uses the same `PromptEvalPoller` (it emits via its module-level `_emit_phase`, which also merges section/chunk structural state). During a **dark window** (cold model load, or a build-9010 prompt-eval that dropped the `/slots` token counters) the poller heartbeats a ticking `"evaluating prompt (Ns)"` so the label never freezes; a `/slots` probe failure is logged once at WARN (with the resolved URL) then DEBUG. A poller span can wrap several LLM calls (card_gen batches, mechanics council, ai_review per-card) — it restarts its gen clock when the decode counter drops so tok/s is per-call, not a cross-call average.
  2. For synchronous, non-streaming AI tab actions that hold the AI lock but emit no telemetry (e.g. the Mechanics tab's "Re-pick with AI" + candidate refresh), call `MTGAIWizard.showBusy(label)` / `clearBusy()` — same strip, but an **indeterminate** animated bar (`wiz-progress-bar--indeterminate`) instead of a determinate fill. `showBusy()` paints first; once the poller's first tick lands the strip upgrades to determinate tok/s, and `clearBusy()` (always run in the tab's `finally`) tears it down.
  - The AI mutex guarantees these two never overlap. When a coarse stats-less `"running"` label (e.g. mechanics "candidate 3/8", skeleton "attempt 2/3") interleaves with poller tok/s ticks, `handlePhaseEvent` keeps the determinate bar for a short grace window (`_LIVE_STATS_GRACE_MS`) so the bar doesn't strobe between determinate and indeterminate. Reusable by any tab; don't fake a determinate % when there's no real progress signal.

## Local LLMs
Local models run through llmfacade's `llamacpp` provider in managed mode (lazy-spawns `llama-swap`). Registry: `backend/mtgai/settings/models.toml`. For setup, gotchas, and adding a model, see `learnings/local-llm-setup.md`. For benchmarks: `learnings/llamacpp-tc2-benchmark.md`. For the Gemma repetition-loop story: `learnings/gemma-repetition-loops.md`. For context-length tiers: `learnings/ctx-tier-sweet-spots.md`.

**Context-length tiers**: a registry `context_window` is BOTH the llama-server `--ctx-size` (KV pre-allocated at load) AND the token budget, so a downstream stage that never sees a big input wastes KV VRAM at 128k. Only `theme_extract` needs a large window (~58.7k); every other stage is ≤~24k. Tiering is automatic per stage: `get_llm_model_id` (the single seam shared by every generator and the tok/s poller) keeps `theme_extract` on the base's 128k window and resolves every other stage (`_FULL_CONTEXT_STAGES` is the allowlist) to a 48k twin (same GGUF, smaller `--ctx-size`, distinct `model_id`) to free KV → fewer weight layers spill to CPU → faster. Gemma is SWA (only global-attention layers' KV scales with ctx), so the whole 128k→8k KV swing is just ~1.2 GiB and 48k already captures ~72% of it — hence 2 tiers (one swap boundary), not 3. Twins are synthesized in code (`_CONTEXT_TIER_TWINS` in `model_registry.py`) by cloning their base and flagged `internal` so they're hidden from every model-picker — the user chooses a base, the tier is chosen per stage. Profile per-stage maxima from real transcripts with `research/scripts/ctx_log_audit.py`.

**Sampling temperatures** (`generation/temperatures.py`): every LLM call's `temperature` resolves to a named constant in this one module (the `token_budgets.py` of temperatures), so the values are documented in one place instead of scattered magic numbers. A small ladder (`GREEDY` 0.0 / `ANALYTICAL` 0.3 / `BALANCED` 0.7 / `CREATIVE` 1.0) plus tuned outliers (`PRECISE` 0.2, `FOCUSED` 0.4, `GROUNDED` 0.6, `INVENTIVE` 0.9). **Two hard rules, both measured:** (1) `GREEDY`/temp-0 is **cloud-only** — on a local model greedy decode triggers repetition loops *and* llama.cpp bypasses its anti-loop samplers at temp 0, and it isn't actually deterministic anyway; (2) any low-temp local call must pair with a `RETRY_TEMP_STEP` (0.2) per-retry bump — the verified loop escape (`repeat_penalty` is not, for Gemma; see `learnings/gemma-repetition-loops.md`). The review gates (`gate_common`), slot grouper, and reprint selection all do this.

**LLM call logs**: llmfacade logging is on by default — every `generate_with_tool` call writes a per-conversation JSONL **and HTML** transcript, named after its tool via `llm_client._convo_name()` (`select_best_mechanics-<uuid8>.html`, etc.) so logs are identifiable. Pass `generate_with_tool(log_dir=<dir>)` to route the transcript: the generators point it at their stage's `<asset>/<stage>/logs` (mechanics → `mechanics/logs`, archetypes → `archetypes/logs`, visual_refs → `art-direction/logs`, card_gen → `card_gen/logs`) so the transcript sits beside the stage artifacts. A Path writes flat into that dir (no `llmfacade<stamp>` session subdir, no auto-prune); omitting `log_dir` (default `None`→`True`) falls back to llmfacade's session dirs under `backend/logs/`. This llmfacade transcript replaced the hand-rolled per-generator logging (don't reinvent it). `card_generator` is the exception: it keeps its bespoke per-card/batch log for the post-generation **validation errors + applied fixes + cost** llmfacade can't capture, *and* routes the transcript to the same dir.
