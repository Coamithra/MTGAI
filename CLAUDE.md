- Python 3.12+ (uv-managed). Lint: `ruff check .` / `ruff format .` from `backend/`. Tests: `pytest` from `backend/`.
- **Before starting the server, pull latest `master` first.** Whenever the user asks to start/run the server (`python -m mtgai.review serve --open`, the `/run` skill, etc.), run `git pull` (fast-forward) on `master` *before* launching ‚Äî fixes land on `master` from parallel worktree branches, so a stale local checkout makes the user re-report bugs that are already fixed. ## QA / debug harness
- **`serve --debug` (`MTGAI_DEBUG=1`)** mounts the QA/debug surface for the self-driving **QA Bot** (`/qa-bot` skill). Off by default ‚Äî invisible in a normal run. It mounts `mtgai/pipeline/debug_routes.py` (`attach_debug_routes`, gated by `is_debug_enabled()`) and injects the floating üêõ **QA Debug** panel (`gallery/templates/static/wizard_debug.js`, included by `wizard.html` only when the `debug_enabled` template flag is set in `_render_wizard`).
- **Why it exists**: a browser agent (claude-in-chrome) can't touch the native OS save/open picker (`window.showSaveFilePicker()` is invisible to the DOM) and can't afford the slow early pipeline phases. The debug endpoints route around both:
  - `POST /api/debug/quick-project` ‚Äî create + activate a QA project **server-side** (no picker), applying the **`qa` preset** (every LLM stage ‚Üí `gemma4-26b-iq2m`, thinking `disabled`; in `model_settings.PRESETS`) + small `set_size` + `use_prefab_{cards,mechanics}` on.
  - `POST /api/debug/seed-stage {target_stage, source_dir?}` ‚Äî clone a finished "golden" project (auto-detected, or `MTGAI_QA_GOLDEN`) into `output/qa-runs/<name>/`, rewrite `pipeline-state.json` so stages ‚â§ target are COMPLETED and the rest reset PENDING, and land the wizard on that tab with real upstream artifacts present. The "skip to a state" debug jump.
  - `POST /api/debug/open-path {path}` ‚Äî open a `.mtg` from a server path (no picker); `POST /api/debug/save-mtg` ‚Äî write the active project's `.mtg` to its asset folder (the Save-button bypass `wizard_project.js` routes to when `window.MTGAI_DEBUG`).
- **The `/qa-bot` skill** (`.claude/skills/qa-bot/`) is the methodology: one serial QA orchestrator drives the app via claude-in-chrome (single live app + AI mutex ‚áí serial driving), files each confirmed bug to Trello (`bug` label), and farms the fix to a subagent that runs `CONTRIBUTING.md` end-to-end (full auto self-merge) in its own worktree (code fixes parallelize even though driving doesn't); it restarts the server + hands off to a fresh orchestrator once enough fixes have merged.

## Code Style
- Pydantic v2 for all data structures. StrEnum for enumerations.
- `X | Y` union syntax (not `Union[X, Y]`). `list[X]` (not `List[X]`).
- Line length 100. Imports sorted by Ruff (isort).

## Data Models
- Card schema is the single source of truth: `mtgai/models/card.py`. Field names match Scryfall (`oracle_text`, not `rules_text`).
- Card data: JSON files in `output/sets/<SET_CODE>/cards/`.
- Naming: `<collector_number>_<card_name_slug>.json`; art: `..._v<attempt>.png`; render: `....png`.
- **Repo-relative paths**: never hardcode an absolute path like `Path("C:/Programming/MTGAI/...")` ‚Äî it breaks on any other checkout / CI. `mtgai/io/paths.py` is the canonical seam: `repo_root()` (the repo root, via `Path(__file__).resolve().parents[3]` from `mtgai/io/`) and `output_root()` (`repo_root()/"output"`, the gitignored sibling of `backend/`). Library code imports these; standalone `backend/scripts/*.py` (not on the package path) inline `Path(__file__).resolve().parents[2]`. A file's own bundled assets (prompt dirs, art workflows) use `Path(__file__).resolve().parent / ...`. External tool roots stay machine-specific but read from env where trivial (e.g. `COMFYUI_ROOT` env var, default `C:/Programming/ComfyUI`).

## Architecture overview

The codebase is a multi-stage pipeline that turns a setting prose document into a finished MTG set (cards + art + rendered images). Stages are orchestrated by `mtgai/pipeline/` and surfaced at `/pipeline` in the local FastAPI server (`python -m mtgai.review serve --open`). Each stage is independently runnable from the CLI. The toolchain is set-agnostic; the project is driven by the wizard's per-project `.mtg` flow.

Card statuses: `draft ‚Üí validated ‚Üí approved ‚Üí art_generated ‚Üí rendered ‚Üí print_ready`. Pipeline is resumable; every generation attempt is tracked.

**Re-entrant pipeline / review‚Üíregen loop**: a stage can appear more than once.
- The post-`card_gen` review **gates** (the merged Conformance & Interactions gate ‚Äî two internal LLM steps ‚Äî then Design Review) check card-vs-slot conformance, whole-pool interactions, and design quality; on failure they stamp `regen_reason`/`flagged_by` onto the offending cards and set `StageResult.rerun_from="card_gen"`.
- The engine then *appends a fresh instance span* `[card_gen ‚Ä¶ that gate]` right after the flagging step and walks forward into it (forward-only, never a backward jump).
- `StageState.instance_id` is the per-instance/tab id ‚Äî `== stage_id` for the **backbone**, `f"{stage_id}.{n}"` (e.g. `conformance.2`, displayed "Conformance & Interactions 2") for inserted copies; the shared `stage_id` still keys the runner, clearer, model assignment, and break-point. Each instance is its own wizard tab.
- The review‚Üíregen loop is **capped per gate** (`engine._handle_rerun`): a gate that flags cards bounces to `card_gen` for another round until they conform, but once *that gate* has run its budget of regen rounds a further flag from it no longer bounces ‚Äî the flagging gate completes and its still-flagged cards are accepted as-is so the pipeline can finish (a safety valve against a never-conforming card, NOT a quality bar). The budgets are **separate** so conformance churn can't starve the later council: `MAX_REGEN_ROUNDS` (5) for the **conformance** gate, `MAX_AI_REVIEW_REGEN` (2 ‚Üí 3 total council passes) for **ai_review**. Each gate's rounds-so-far is derived from the plan, not a stored counter ‚Äî every bounce inserts one `card_gen` + one `conformance` instance and only an ai_review bounce also inserts an `ai_review`, so `ai_review rounds = N_ai_review ‚àí 1` and `conformance rounds = N_card_gen ‚àí N_ai_review`; the cap checked is the one for the gate that actually flagged. Total `card_gen` is bounded by `1 + MAX_REGEN_ROUNDS + MAX_AI_REVIEW_REGEN`. (The only *pause* mechanism on a gate is still an explicit "stop after this step" break-point.)
- A stage's review pause is decided from the project's **live** `break_points`, re-read the moment the stage finishes (NOT the build-time-frozen `review_mode`), so a "Stop after this step" toggled *while the stage is running* takes effect on the stage that just ran. Only **backbone, review-eligible** instances (`instance_id == stage_id` and `review_eligible`) honour the live toggle; inserted regen-loop copies stay pinned AUTO so an active loop still flows, and a **review-ineligible stage is forced AUTO at the pause check** (`if not stage.review_eligible: review_mode = AUTO`) so it can **never** pause ‚Äî a stale/erroneous persisted `review_mode=REVIEW` (e.g. a debug-seeded clone that inherited "review" from its golden source) can't sneak it past the contract (`engine._live_break_point` + the `_run_loop` pause check, which also syncs `review_mode` so persisted state stays honest). The wizard mirrors this: `wizard_stage.js` `breakPointToggleHtml` hides the "Stop after this step" toggle on a `review_eligible === false` stage tab (a control the engine won't honour). The debug seed (`debug_routes.debug_seed_stage`) also re-resolves every backbone stage's `review_mode` from the QA project's live `break_points` so a cloned golden never seeds stale modes.
- `card_gen` treats a slot whose card has `regen_reason` as needing regeneration: it archives the prior card, threads the reason into the prompt, and clears the flag on the new card.
- **Per-instance card-pool version tracking** (`pipeline/history.py`): the live `<asset>/cards/` folder stays the single working set (always the loop tip), but on completion each loop instance (`lands` + `card_gen` + the `conformance` & `ai_review` gates = `SNAPSHOT_STAGES`) snapshots the live `cards/*.json` + `generation_progress.json` to a write-once `<asset>/history/<instance_id>/` at the engine `save_state` seam (`_snapshot_instance_output`; `_regen_archive/` excluded; plain copies). That inter-stage seam ‚Äî the snapshot **plus** the `_handle_rerun` regen-span insertion ‚Äî runs under a brief engine-held `ai_lock` re-acquire (the runner just released its own hold), released again before the next runner takes its own hold (the lock is strictly non-reentrant); if a guarded_ai endpoint wins the sub-ms reacquire race the engine skips just the snapshot (degrades to a from-live re-run) rather than copy a pool in flux, while the span insertion (engine-owned state) always runs. An instance's *entry* pool is its predecessor's snapshot, pinned as `StageState.entry_snapshot_id` when it starts. `engine.rerun_instance(state, id)` re-runs any duplicable instance (`RERUNNABLE_STAGES` = the three loop stages) from that entry: restore the snapshot ‚Üí truncate downstream (+ their history) ‚Üí reset the instance ‚Üí re-append the canonical forward path via `_build_forward_path` (the *forward* mirror of `_build_rerun_span`, auto-numbered so a surviving sibling yields `conformance.2` while a fresh stage yields backbone `ai_review`; loop stages stay AUTO, post-loop stages honour break points) ‚Üí set `NOT_STARTED` so the caller's `_kickoff_pipeline_engine` walks forward from it. Surfaced by `POST /api/wizard/instance/rerun {instance_id}` (409 if engine/extraction running; same cascade contract as `edit/accept`) + the per-tab "Re-run this step" button (`W.rerunButtonHtml`/`bindRerunButton`/`rerunInstance` in `wizard_util.js`). Tab card-views read `history/<id>/` for a completed *non-tip* instance and live `cards/` for the tip (`?instance_id=` on `card_gen/state` ‚Üí `_resolve_view_cards_dir`), fixing the stale "old tab shows the latest cards" view. Pre-version-tracking projects (no `entry_snapshot_id`/`history/`) disable the button and degrade to a from-live re-run. **Distinct from rerun: the FAILED-stage retry** (`POST /api/wizard/instance/retry`, optional `{instance_id}` ‚Üí defaults to the current/only failed stage) exposes `engine.retry_current()` ‚Äî reset the failed stage PENDING‚ÜíRUNNING and re-run it **in place**, NON-destructively (no entry-snapshot restore, no downstream truncation, no cascade confirm), relying on each stage's resume-skip so finished work is kept (art_gen skips generated cards, card_gen skips filled slots, finalize re-processes). It's the UI for recovering a stage that ended FAILED (engine crash, a cancel returning `success=False`, or an orphaned RUNNING stage demoted to FAILED by `cleanup_orphan_running_stages` on reopen ‚Äî which would otherwise need manual pipeline-state.json surgery, since the heal‚Üíadvance path *skips* a stage whose work didn't finish). Surfaced as a "Retry this step" button in the shared stage-tab header chrome (so it shows on EVERY failed tab, including the custom-rendered finalize/art_gen/rendering that lack an AI Refresh) + a primary button on the stage-failure modal (`W.retryButtonHtml`/`bindRetryButton`/`retryStep` in `wizard_util.js`); 409 if the engine or extraction is running. The Card Generation tab uses this same machinery for its progress + old/new view of a regen instance: it scopes the "X / Y cards generated" headline to the slots THAT instance (re)creates (`generate_set` reports `run_target = len(unfilled)`, not the full set, so a regen instance reads "5 / 5" not "5 / 277"), and highlights the cards an instance regenerated vs carried over by diffing its pool against its `entry_snapshot_id` (`/api/wizard/card_gen/state` returns `is_regen_instance` + a per-card `is_new`; the per-card SSE stream marks streamed cards `is_new=True`; the highlight is suppressed on the first `card_gen` / pre-version-tracking projects, where "everything is new" is not a useful distinction).
- The generic edit-cascade (`_apply_cascade_clear`) clears card_gen via the **scoped** `clear_card_gen` ‚Üí `clear_card_gen_cards()` (card_gen-owned `cards/*.json` only, plus `generation_progress.json` + `_regen_archive/`, **preserving the lands stage's `L-*`** ‚Äî shared with the `card_gen/refresh` endpoint via `stages._is_land_stage_card`), and drops the cleared instances' `history/` snapshots; the rerun path above is the precise per-instance reset that restores entry snapshots instead of deleting.
- **Two past-tab edit models** (`plans/wizard-tab-conventions.md` ¬ß6). A completed **stage** tab's "Edit" is the **unlock** path: `_apply_downstream_clear(state, idx)` + `POST /api/wizard/edit/unlock {from_stage}` clears `stages[idx+1:]` (artifacts + history + regen-inserted duplicate instances), **keeps** the edited stage's output, and sets it `PAUSED_FOR_REVIEW` (pipeline `PAUSED`) so it becomes the active, editable, restartable tip (single "this deletes every tab after this one" confirm ‚Äî no draft/revert). The **content** tabs (Theme, Project Settings) keep the deferred draft flow through `_apply_cascade_clear` + `/edit/accept`, which clears `stages[idx:]` *including* stage 0 and re-runs (cheap-to-revert form fields, and editing them invalidates the whole pipeline). `_compute_cascade_preview(..., after_only=True)` lists downstream-only for the unlock modal. Editing of any kind is gated on no LLM running (`isPipelineRunning`); a completed stage's model assignment is freely changeable mid-run **except** for the currently-running stage (`_stage_is_running` ‚Üí 409 in `wizard_project_save_model`). On stage completion the next tab **auto-opens** only when the user is viewing the tip (`updateStageStatus`); pure view controls (sort/filter) stay live during a run.

**Key modules** (look in the file for detail; CLI flags via `--help`):

- `mtgai/pipeline/` ‚Äî end-to-end orchestrator: stage registry, FastAPI server, SSE event bus, wizard shell. State persisted to `<asset_folder>/pipeline-state.json` (the asset folder the user picked on Project Settings).
- `mtgai/pipeline/wizard.py` ‚Äî wizard state resolver. Maps the on-disk `pipeline-state.json` + `theme.json` into the visible-tabs / latest-tab shape `wizard.html` consumes. URL routing lives in `pipeline/server.py` (`/pipeline/<tab_id>`). The legacy `/pipeline/configure` route is a theme-aware redirect into the wizard.
  - Project Settings (kickoff) tab payload comes from `GET /api/wizard/project`; live-apply edits go through `/api/wizard/project/{params,theme-input,breaks,models,preset/apply,preset/save}`; `POST /api/wizard/project/start` validates + spawns the theme extractor and tells the client to navigate to `/pipeline/theme`. The extraction worker, when launched via the kickoff path, also writes the assembled theme.json on `done` so the Theme tab finds populated content on next mount.
  - **Building or modifying a wizard tab? Read `plans/wizard-tab-conventions.md` first** ‚Äî it documents the patterns Project Settings + Theme establish (single Save & Continue footer button, required-field asterisks, form lock during AI gen, AI provenance badges, past-tab edit cascade, streamed-LLM textareas, etc.) so new tabs stay consistent.
- `mtgai/pipeline/theme_extractor.py` ‚Äî theme content extraction. Surfaced as the Theme tab inside the wizard. Uploads PDF/text, runs multi-stage LLM extraction, then JSON pass for constraints + card suggestions.
- `mtgai/generation/llm_client.py` ‚Äî all LLM transport routes through [llmfacade](https://github.com/Coamithra/LLMFacade) (Anthropic + llama.cpp). Single entry point `generate_with_tool()`; provider resolved per-call from the model registry.
  - **Prompt caching primitive**: pass keyword-only `system_blocks=[(text, cache), ...]` (each item a plain `str` = uncached, or a `(text, cache)` tuple) instead of `system_prompt` to place bulk STATIC context in its own independently-cached Anthropic system block, and/or `cache_user=True` to cache the last user block. Anthropic caps a request at **4 `cache_control` markers** (tools + system blocks + optional last user), so bundle all static context into **ONE** block. Both are no-ops on llamacpp (blocks flattened to one joined system string, cache flags dropped, `cache_*` stay 0). `card_gen` uses it via `prompts.build_static_set_context()` ‚Äî the ~6-7k-token set-context prefix (setting prose + mechanics + archetypes + preventive guidance) rides in a cached system block hoisted out of the batch loop, written once and read at ~0.1x across the ~200-250 per-set batches; the dynamic per-batch content (existing-card context, cycle siblings, slot specs) stays in the user message via `build_user_prompt()`. The mechanics council's same-round reviewers use `cache_user=True` (byte-identical user prompt). The structural stages follow the same split (static set context in one cached system block, dynamic trigger/data in the user turn): `skeleton_knobs_tuner` (context = `skeleton_knobs_context.txt` formatters), `reprint_selector` select (context = set framing + the ~350-card pool) and place (context = chosen reprints + slot list), and `land_generator` basics + investigation. `archetypes` and the skeleton `relabel` pass already keep their static context in the system prompt (auto-cached), so they were audit-only. These are low call-volume (1-6 calls/stage), so the win is consistency + retry/re-roll reuse within the TTL, not the per-set cost win.
- `mtgai/settings/` ‚Äî model registry (`models.toml`) + the active project's `ModelSettings` (in-memory only; the .mtg file is the persistent store) + global default-preset (`output/settings/global.toml`).
  - The settings shape carries `[llm_assignments]` / `[image_assignments]` / `[effort_overrides]` / `[break_points]` / `[set_params]` (set_name + set_size + mechanic_count) / `[theme_input]` (kind + filename + upload_id + char_count + uploaded_at) / `asset_folder`.
  - All resolution goes through `get_active_settings()` (or `require_active_project().settings`); helpers no longer take `set_code`. Live-apply endpoints call `apply_settings(new)` to update the pointer; the user saves the .mtg via the browser's File System Access API to persist.
  - Profiles save with `profile_only=True` so they exclude `set_params` + `theme_input` (per-project, not template-able). Cross-set defaults UI at `/settings`; per-project Project Settings UI is the wizard's first tab (`/pipeline/project`, endpoints under `/api/wizard/project/*`).
- `mtgai/validation/` ‚Äî two-tier validators (AUTO auto-fixed; MANUAL becomes LLM retry prompts). Cards immutable; fixers return new instances. The `validate_card` runner sequence is schema ‚Üí whitespace ‚Üí blank_lines ‚Üí mana ‚Üí type_check ‚Üí rules_text ‚Üí keyword_casing ‚Üí keyword_ordering ‚Üí text_overflow ‚Üí uniqueness. `whitespace` (`validation/whitespace.py`) is an AUTO check that runs FIRST: local models sometimes double-escape newlines in JSON tool output, leaving a LITERAL backslash-n (or backslash-t) in `oracle_text`/`flavor_text`; `normalize_escaped_whitespace` is the one canonical fix (literal backslash-n ‚Üí real newline, literal backslash-t ‚Üí space), applied before the line-based fixers below so they see real line structure. `persist_mechanic_selection` reuses the same function for mechanic example cards (which never pass through `validate_card`), and `rendering/text_engine.py` keeps its render-time replace as the backstop. `keyword_ordering` (`validation/keyword_ordering.py`) is an AUTO check: keyword-ability lines (evergreen + the set custom mechanics, via `rules_text.all_keywords()`) must sit **above** complex triggered/activated/static abilities; the fixer stable-partitions keyword lines to the top, moving each line whole so injected `(reminder)` text rides along untouched (per the validators-skip-parenthesized-text contract). The custom-keyword vocabulary is resolved per-call from the active project's `mechanics/approved.json` mechanic names (`rules_text.custom_keywords()`, cached by path+mtime; `set_custom_keywords()` overrides it for tests/callers), so the validators recognize **any** set's keywords ‚Äî not a hardcoded seed. **Canonical mana-cost ordering** (`mana`, card 6a29ba2c): the `mana.wubrg_order` AUTO check + `fix_wubrg_order` reorder a cost's symbols to canonical print order via `validation/mana.py:_canonical_symbol_order` ‚Äî generic/X/`{C}` first, then colored pips in the order `canonical_color_sequence(colors)` returns: the **shortest contiguous arc of the WUBRG colour wheel** ("shortest way around"), NOT a flat WUBRG sort. This is what real cards print, e.g. `{G}{W}` (not `{W}{G}`) for green-white, `{R}{W}` for red-white, `{G}{U}` for green-blue ‚Äî the three wheel-wrap pairs (`GW`/`RW`/`GU`) are exactly where the old naive WUBRG sort was wrong (it mangled them to `WG`/`WR`/`UG`, so the title-bar cost disagreed with the two-colour split frame). Two-colour = the wheel pairs; three-colour = the ten shards (contiguous arc) + wedges (center color in the middle, flanked pair clockwise ‚Äî the Khans `+2`-step convention, e.g. Jeskai `{U}{R}{W}`); four-colour = arc-minus-one-gap; five-colour = `WUBRG`; mono trivial ‚Äî all verified against Scryfall. Distinct from the `colors`/`color_identity` *list* fields, which keep Scryfall's flat WUBRG sort. **Conservative skip:** a cost containing a compound symbol (hybrid `{G/U}` / phyrexian `{G/P}` / twobrid `{2/W}`) is left UNTOUCHED (`cost_has_compound_symbol`) ‚Äî the check never flags it and the fixer is a no-op, so a hybrid pip's perceived semantics are never reordered. **Oracle-text polish AUTO checks** (card 6a29886f): `blank_lines` (`validation/blank_lines.py`, `collapse_blank_lines`) collapses runs of 2+ newlines (empty paragraphs the local model emits between abilities) to one and strips leading/trailing newlines ‚Äî another line-structure normalizer, healed in the SAME `prenormalize_card_whitespace` pre-pass as whitespace and run AFTER the escape-decode so a literal-backslash-n-backslash-n that decoded to a real blank line is also collapsed. `keyword_casing` (`validation/keyword_casing.py`, `fix_keyword_casing`) lowercases a keyword capitalized MID-SENTENCE in running text ("gains Indestructible" ‚Üí "gains indestructible") ‚Äî the running-text complement to `rules_text.fix_keyword_capitalization` (which only fixes the 2nd+ entries of a comma-separated keyword-LIST line); conservative: skips reminder/quoted spans, line/sentence starts, and a keyword adjacent to another capital (likely a proper name), and only single-word keywords. `rules_text.modal_emdash_period` (`fix_modal_emdash_period`) strips a stray period after a modal em-dash ("Choose one ‚Äî." ‚Üí "Choose one ‚Äî"); its ROOT CAUSE was `fix_line_periods` appending a period to a line ending in "‚Äî", now guarded (both the `line_period` check + fixer skip em-dash-terminated lines). The card-gen system prompt (`research/prompt-templates/system-prompt-v1.md`) also instructs the model to template keywords first, but the validator is the durable backstop against prompt drift. **Leaked-P/T recovery** (`type_check`): the local card-gen model habitually writes a creature's stats as a bare `N/N` line in `oracle_text` instead of populating the structured fields (sometimes omitting them entirely). `type_check.pt_in_oracle` is an AUTO check (`find_bare_pt_line` / `fix_pt_in_oracle`, regex `BARE_PT_LINE_RE`) that moves a standalone `N/N` line into `power`/`toughness` (filling only empty fields ‚Äî a real structured value wins) and strips it from the rules box; it fires even when the stats are already set, so a redundant `2/2.` never renders. `rules_text.line_period` (validator + `fix_line_periods`) skips bare `N/N` lines so it can't cement a period onto the leak. When the stats are genuinely absent **and** unrecoverable (no leaked line), `type_check.creature_missing_pt` is now a **regen trigger** (in `_REGEN_TRIGGER_CODES`) so a P/T-less creature is retried/flagged rather than silently saved as `draft`.
- `mtgai/generation/reminder_injector.py` ‚Äî reminder text is **never LLM-generated**, always injected programmatically from mechanic definitions after review.
- `mtgai/review/finalize.py` ‚Äî post-review: inject reminder ‚Üí validate ‚Üí auto-fix ‚Üí save. Writes a JSON sidecar `<asset>/reports/finalize-report.json` (alongside the markdown report) that the **Finalization wizard tab** reads to badge auto-edited cards + show the before/after of what the stage changed (`original_oracle_text` is kept per modified card). - `mtgai/analysis/` ‚Äî the **merged Conformance & Interactions gate** (stage_id `conformance`, display "Conformance & Interactions"), run by `stages.run_conformance` as two LLM steps plus a purely-algorithmic duplicate scan (no LLM) that runs first and folds its hits into the conformance checklist, all under one AI-lock hold + one poller (the two LLM steps resolve the shared `conformance` model assignment):
  - `conformance.py` ‚Äî each card vs. its slot's `tweaked_text` spec; holistic, **no descriptor parser**; the gate's first step. **Batched, flag-only, streamed** (NOT one-call-per-card -- that was correct but unusably slow under the serial AI mutex, ~277 calls/set, and routinely truncated on per-card CoT): the pool is grouped into batches of `BATCH_SIZE` (20 ‚Äî small so the local model's reasoning space per call stays tractable and terminates) and each batch is one streamed `stream_text` call where the model emits a `--CARD <slot_id>--` block (parsed live, like the skeleton relabel) **only for non-conforming cards**. A flag streaming in advances an "approved frontier" (earlier unflagged cards flip to conforming); a clean batch return approves its rest; a truncated batch retries (temp-bumped off the local-reasoning floor) and its unreached tail is left `conforms=None` (unknown, never flagged). ~14 calls/set instead of ~277.
  - `interactions.py` ‚Äî degenerate-combo scan naming the enabler card + a `replacement_constraint`; the gate's second step. **Batched, flag-only, streamed** like conformance, but over **cumulative-context batches**: each batch (`BATCH_SIZE` 20 NEW cards ‚Äî small to keep the new-vs-pool reasoning tractable) is shown all previously-reviewed cards as existing context plus its new cards, and flags every degenerate interaction the new cards create with the existing or other new cards. Each ordered pair is thus checked once (in the later card's batch), so coverage is complete in O(batches) streamed calls instead of one whole-pool call. The enabler may be a new OR existing card; the flagged block carries the diagnosis plus an `AVOID:` line (the `replacement_constraint`). Streams a per-card interaction verdict (`interaction_card`) for the tab's second per-card checkbox. **Precision + attribution** (`INTERACTION_SYSTEM_PROMPT`): the prompt separates *degenerate/unbounded* interactions from *strong-but-bounded* cards ‚Äî it must **respect rate limiters** ("activate only once each turn", sorcery-speed, tap-without-untap, one-shot costs) before claiming a loop and name the exact repeatable cycle, and it weighs **rarity** (a strong tutor/engine/payoff at rare/mythic is acceptable, not a flag) ‚Äî so a once-per-turn engine or a powerful mythic is no longer mislabeled an "infinite combo". It also attributes the flag to the **root-cause enabler**, which is *often an earlier (existing-context) card*, NOT the new card under review ‚Äî the new card is frequently an innocent payoff. Because flags key on `enabler_slot_id` (validated against every gate card) and `_flag_cards_for_regen` + `card_gen._collect_flagged_slots` act on any slot regardless of order, flagging an earlier enabler already re-threads the `card_gen` bounce correctly (no engine change); the conformance tab's `interaction_card` handler shows such an earlier enabler's Conformance column as muted `¬∑` since it wasn't conformance-checked on a scoped re-run.

The server endpoints persist the active project's `.mtg` body (the TOML *content*, since the browser ships content not a path via the File System Access API) to `output/supervisor/last-project.mtg` while running as a supervised child (`_persist_supervised_project` in `pipeline/server.py`, on `/api/project/{open,materialize}`; cleared on `/api/project/new`). The supervisor clears that file + the retry-ceiling counter at session start and flags only a **restart** child (never the first spawn) with `MTGAI_AUTO_RESUME=1`. On such a boot the child lifespan calls `auto_resume.start_auto_resume()` ‚Üí re-opens that project server-side (`parse_project_toml` + `write_active_project` + `cleanup_orphan_running_stages`, the in-process equivalent of `/api/project/open`), finds the orphaned-then-FAILED stage, and re-fires `engine.retry_current` via `pipeline/server.py:_spawn_retry_engine` (the same engine-spawn helper the manual `/api/wizard/instance/retry` endpoint uses). **Crash-resume loop guard** (pairs with the fast-failure boot-loop guard): a **progress-aware per-stage ceiling** (`auto_resume.decide`, `RESUME_CEILING` 3) ‚Äî a stage that crashes while `progress.completed_items` *advanced* since the last attempt (the bounded art_gen kill, carried by resume-skip) resets the counter so a healthy run resumes indefinitely, while a stage that crashes *without* progress is a poison pill, counted and left FAILED for a manual retry after the ceiling. State persists in `output/supervisor/auto-resume-state.json`. All opt-in ‚Äî a normal `serve` is unaffected (no heartbeat, no supervisor, no persistence).

## Cross-cutting

- **AI mutex (`mtgai/runtime/ai_lock.py`)**: app-wide, **one AI call at a time**. All AI-touching endpoints share it. Conflicting requests get 409 with a busy payload. Wizard refresh/regenerate endpoints wrap their AI work in `pipeline/server.py`'s `with guarded_ai("Action name", stage_id=‚Ä¶) as guard: if guard.busy: return guard.busy_response` ‚Äî the context manager owns acquire‚Üí409 + the `try/500 {"error": ‚Ä¶}` envelope (worker exceptions become `AIActionError`, rendered by the registered handler) + the success-only `_heal_failed_stage(stage_id)` + release. Suppress the heal after a cancelled run with `guard.skip_heal = True` (the card_gen contract). The engine path (`stages.py`) still uses `ai_lock.hold(...)` directly. The single busy-409 helpers are `_busy_response()` and `_reject_if_busy()` (the latter for gate-only endpoints that don't hold the lock ‚Äî theme section-refresh / kickoff). Long-running workers (card_gen, mechanics, lands, reprints, skeleton relabel, conformance, ai_review, art_prompts, art_gen) poll `ai_lock.is_cancelled()` at their loop/call boundaries and stop early (keeping partial output) so the Cancel button actually halts them; the guard then skips the heal. **Strictly non-reentrant** (NOT an `RLock`): a thread already holding the lock that calls `hold()`/`try_acquire()` again gets busy back (`None`) and the nested body is silently skipped ‚Äî acquire once at the outermost layer. `try_acquire()`/`hold()` return a monotonic `run_id` (`int` >= 1 on success, `None` when busy; truthy/`is not None` checks both work); `request_cancel(run_id=...)` only cancels if the active run matches that id, so a stale cancel can't kill a newer run that reused the lock. `request_cancel()` with no arg keeps the legacy "cancel whatever's running now" semantics (used by the UI cancel button + project-switch drain).
- **Tab state (`mtgai/runtime/runtime_state.py`, `extraction_run.py`)**: server is the source of truth for AI run state, pipeline state, and saved theme.json; browser holds UI ephemera in localStorage. SSE streams are broadcastable (tab-switching mid-run does not cancel; cancel is opt-in via the explicit button). New long-running AI runs should follow the same broadcast pattern. Aggregator: `GET /api/runtime/state`.
- **Active project**: which `.mtg` is open is the source of truth. The server holds a process-memory `ProjectState` pointer (`mtgai/runtime/active_project.py` ‚Äî set_code + settings + .mtg path) ‚Äî set by `/api/project/{open,materialize}`, cleared by `/api/project/new`, lost on restart so a fresh boot always greets the user with New / Open. Stage outputs land under the project's `asset_folder`; `mtgai/io/asset_paths.set_artifact_dir()` reads the active project and raises `NoAssetFolderError` (translated to 409 at endpoints) if none is open. Switching projects mid-AI-run requires `force=true` in the request body; without it the endpoint returns 409 + the AI lock's busy payload so the client can render a confirm modal. The cancel path calls `ai_lock.request_cancel()` then `active_project.await_lock_release()` before flipping the pointer. **Every project switch (`/api/project/{open,new,materialize}`) also calls `server._reset_engine()`** ‚Äî the module-level `_engine`/`_engine_task` must NOT outlive their project. `_get_current_state()` prefers `_engine.state` whenever an engine exists (even an idle one whose run finished), so a surviving reference would serve the *previous* project's state from every state-reading endpoint and ‚Äî worse ‚Äî let the first `_heal_failed_stage` resolve A's stale state and `save_state` it into B's `pipeline-state.json` (the path resolves through the *active* project, B), clobbering B's config with A's. `_reset_engine` runs after `_project_switch_guard` has drained any in-flight run, so the daemon thread/task is already winding down and nothing live is lost.
- **Reminder text**: not validated, not LLM-generated. Stripped + injected by `reminder_injector` after review. Validators that touch oracle text must skip parenthesized text.
- **Progress UI**: the wizard's **global progress strip** (`#wiz-progress-strip` in `wizard.js`, sticky at top) is the one canonical progress surface. Driven two ways:
  1. SSE `phase` events on `/api/pipeline/events` ‚Üí `handlePhaseEvent` ‚Äî fed by `StageEmitter.phase()` for engine stages and `event_bus.stage_phase()` for tab-refresh. **Live token-rate stats** (prompt-eval / generation tok/s) come from the shared `PromptEvalPoller` (`generation/phase_poller.py`), which polls llama-server's `/slots` on a daemon thread and emits `generation`/`prompt_eval` ticks. **Every local-model stage is wrapped**, both paths: engine runners (`stages.py`) via `make_poller(stage, emitter.phase, ‚Äî)`, and tab-refresh endpoints (`server.py`) via the `_bus_poller(stage_id, ‚Äî)` context manager (routes the poller through `event_bus` + emits a terminal `phase:"done"` so a trailing tick can't leave the strip stuck). `make_poller` resolves the stage's assigned model and no-ops (`NullPoller`) for a cloud model / no open project ‚Äî so Anthropic stages just get the indeterminate bar. `theme_extractor` uses the same `PromptEvalPoller` (it emits via its module-level `_emit_phase`, which also merges section/chunk structural state). During a **dark window** (cold model load, or a build-9010 prompt-eval that dropped the `/slots` token counters) the poller heartbeats a ticking `"evaluating prompt (Ns)"` so the label never freezes; a `/slots` probe failure is logged once at WARN (with the resolved URL) then DEBUG. A poller span can wrap several LLM calls (card_gen batches, mechanics council, ai_review per-card) ‚Äî it restarts its gen clock when the decode counter drops so tok/s is per-call, not a cross-call average.
  2. For synchronous, non-streaming AI tab actions that hold the AI lock but emit no telemetry (e.g. the Mechanics tab's "Re-pick with AI" + candidate refresh), call `MTGAIWizard.showBusy(label)` / `clearBusy()` ‚Äî same strip, but an **indeterminate** animated bar (`wiz-progress-bar--indeterminate`) instead of a determinate fill. `showBusy()` paints first; once the poller's first tick lands the strip upgrades to determinate tok/s, and `clearBusy()` (always run in the tab's `finally`) tears it down.
  - The AI mutex guarantees these two never overlap. When a coarse stats-less `"running"` label (e.g. mechanics "candidate 3/8", skeleton "attempt 2/3") interleaves with poller tok/s ticks, `handlePhaseEvent` keeps the determinate bar for a short grace window (`_LIVE_STATS_GRACE_MS`) so the bar doesn't strobe between determinate and indeterminate. Reusable by any tab; don't fake a determinate % when there's no real progress signal.
- **In-app dialogs (`gallery/templates/static/wizard_dialog.js`)**: the wizard NEVER uses native `window.confirm/alert/prompt` ‚Äî those are OS-level windows outside the page DOM that freeze page JS, so a single one hangs the claude-in-chrome QA driver (it can only reach the DOM); they're also off-theme. Use the Promise-based `window.MTGAIDialog.{confirm(msg,opts?)‚Üíbool, alert(msg,opts?)‚Üívoid, prompt(msg,default?,opts?)‚Üístring|null}` instead. It's a standalone module (no `MTGAIWizard` dependency, loaded first in `wizard.html`) that reuses the existing `.wiz-modal*` / `.wiz-btn-*` chrome (so it tracks the dark theme for free; only the prompt input + message text are injected once), is focus-trapped (Tab cycles its own controls, Esc = cancel, Enter = confirm/submit, backdrop = cancel) and restores focus on close. The OK / Cancel / input carry stable `data-testid="mtgai-dialog-{ok,cancel,input}"` hooks for automation. Every call site is inside an `async` function so the migration is `confirm(x)` ‚Üí `await window.MTGAIDialog.confirm(x)`. `opts`: `{title, okLabel, cancelLabel}`. (The generic `runAiAction({confirm})` hook in `wizard_util.js` already routes through it, so most tabs get it for free.)
- **In-app image lightbox (`gallery/templates/static/wizard_lightbox.js`)**: the one reusable click-to-zoom pop-out for the wizard's image grids. Same DOM-overlay-not-native-window contract as the dialog module (a native window would freeze page JS + hang the claude-in-chrome QA driver + be off-theme), and standalone in the same way (no `MTGAIWizard` dependency, loaded first in `wizard.html` beside `wizard_dialog.js`). `window.MTGAILightbox.open(src, {alt?, caption?})` (or the object form `open({src, alt, caption})`) overlays the full-size image centered on the `.wiz-modal-overlay` backdrop; `close()` tears it down. Focus-trapped on its close button (Tab stays put, Esc / backdrop click / the ‚§¢ close button all close), restores focus on close, single live instance (a second `open()` replaces the first). Hooks: `data-testid="mtgai-lightbox{,-frame,-img,-close}"`. **Wired on the two image tabs:** Art Generation & Review (`wizard_art_gen.js` ‚Äî a hover-revealed ‚§¢ button per version tile opens the lightbox; the tile click still *picks* the version, the zoom button `stopPropagation`s so it never also picks) and Rendering & Final Review (`wizard_rendering.js` ‚Äî the rendered thumbnail itself is click-to-zoom, `cursor: zoom-in`). Both read the live `<img>.src` so a reroll/re-render's cache-busted pixels show. Use it for any future image grid rather than re-rolling a zoom surface.

## Local LLMs
Local models run through llmfacade's `llamacpp` provider in managed mode (lazy-spawns `llama-swap`). Registry: `backend/mtgai/settings/models.toml`. For setup, gotchas, and adding a model, see `learnings/local-llm-setup.md`. For benchmarks: `learnings/llamacpp-tc2-benchmark.md`. For the Gemma repetition-loop story: `learnings/gemma-repetition-loops.md`. For context-length tiers: `learnings/ctx-tier-sweet-spots.md`.

**Context-length tiers**: a registry `context_window` is BOTH the llama-server `--ctx-size` (KV pre-allocated at load) AND the token budget, so a downstream stage that never sees a big input wastes KV VRAM at 128k. Only `theme_extract` needs a large window (~58.7k); every other stage is ‚â§~24k. Tiering is automatic per stage: `get_llm_model_id` (the single seam shared by every generator and the tok/s poller) keeps `theme_extract` on the base's 128k window and resolves every other stage (`_FULL_CONTEXT_STAGES` is the allowlist) to a 48k twin (same GGUF, smaller `--ctx-size`, distinct `model_id`) to free KV ‚Üí fewer weight layers spill to CPU ‚Üí faster. Gemma is SWA (only global-attention layers' KV scales with ctx), so the whole 128k‚Üí8k KV swing is just ~1.2 GiB and 48k already captures ~72% of it ‚Äî hence 2 tiers (one swap boundary), not 3. Twins are synthesized in code (`_CONTEXT_TIER_TWINS` in `model_registry.py`) by cloning their base and flagged `internal` so they're hidden from every model-picker ‚Äî the user chooses a base, the tier is chosen per stage. Profile per-stage maxima from real transcripts with `research/scripts/ctx_log_audit.py`. **Conformance exception:** the `conformance` gate keeps the base's FULL window (not the 48k twin) once `set_params.set_size >= _CONFORMANCE_FULL_CONTEXT_SET_SIZE` (400), because its interaction step's cumulative-context prompt grows with set size and can outgrow 48k on a large set (the only stage whose input scales with set size). This assumes the assigned base actually has a large window. A **low-context** base that can't honour it no longer degrades silently: `interactions.analyze_interactions` bounds each cumulative batch's existing-context to the assigned model's **actual** `context_window` (via `_bound_existing_context`, mirroring `check_pre_call`'s budget) ‚Äî a roomy window drops nothing (full coverage), a too-small window slides a most-recent window so the prompt never trips `ContextOverflowError` (which `stream_flag_batch` would treat as a failed attempt ‚Üí batch left unchecked), and a single WARN names the model/window/set-size/dropped-count so the reduced far-apart-pair coverage is visible. That server-log WARN is also surfaced to the user: the Project Settings model picker shows an amber notice on the Conformance & Interactions row when the assigned model's effective window can't hold the projected largest interaction batch for the configured `set_size`. `ModelSettings.conformance_context_status()` compares `interactions.project_largest_batch_tokens(set_size, mechanic_count)` (the largest cumulative batch, estimated from synthetic representative cards through the real `_build_batch_prompt`/`count_messages_tokens` so it tracks `_bound_existing_context`) against the **effective** conformance window (the twin, or the base's full window at ‚â•400 via `get_llm_model_id`) with `check_pre_call`'s budget (`int(ctx*(1-SAFETY_MARGIN)) - MAX_TOKENS`). The blob rides in both project payloads and is recomputed by `/api/wizard/project/{models,params}` so the notice live-updates when the conformance model or set_size changes (`conformanceContextNote` in `wizard_project.js`).

**Sampling temperatures** (`generation/temperatures.py`): every LLM call's `temperature` resolves to a named constant in this one module (the `token_budgets.py` of temperatures), so the values are documented in one place instead of scattered magic numbers. A small ladder (`GREEDY` 0.0 / `ANALYTICAL` 0.3 / `BALANCED` 0.7 / `CREATIVE` 1.0) plus tuned outliers (`PRECISE` 0.2, `FOCUSED` 0.4, `GROUNDED` 0.6, `INVENTIVE` 0.9). **Two hard rules, both measured:** (1) `GREEDY`/temp-0 is **cloud-only** ‚Äî on a local model greedy decode triggers repetition loops *and* llama.cpp bypasses its anti-loop samplers at temp 0, and it isn't actually deterministic anyway; (2) any low-temp local call must pair with a `RETRY_TEMP_STEP` (0.2) per-retry bump ‚Äî the verified loop escape (`repeat_penalty` is not, for Gemma; see `learnings/gemma-repetition-loops.md`). The review gates (`gate_common`), slot grouper, and reprint selection all do this. **Local-reasoning anti-loop floor:** an objective/enumeration/judgment stage's *base* temperature is raised to `LOCAL_REASONING_FLOOR` (0.6 = `GROUNDED`) whenever it resolves to a **local reasoning model**, via `temperatures.floor_for_local(base, model_id)` ‚Äî near-greedy decode (PRECISE/ANALYTICAL/FOCUSED) is the single biggest verbatim reasoning-loop trigger on local Gemma 4 (it can spend its whole output budget re-deriving the same CoT and never emit the answer; see `learnings/reasoning-budget-overrun.md` + the "Tame Gemma 4 local overthinking" card). It floors the *base* so the per-retry `RETRY_TEMP_STEP` bump still stacks on top (0.6‚Üí0.8‚Üí1.0); cloud models keep their precise low temp (the lookup is provider-aware + best-effort). Applied at every local low-temp reasoning stage: the conformance + interactions gates (`gate_common.stream_flag_batch`/`generate_gate_tool`), the slot grouper, reprint selection, the mechanics council reviewers + synth + best-of-N pick, and the character-portrait entity scan. The two gates also use a small `BATCH_SIZE` (20) so each call's reasoning space stays tractable and terminates. **DRY-sampler retry escalation (gates only):** on a gate batch's truncation *retry* (`gate_common.stream_flag_batch` / `generate_gate_tool`, attempt >= 2), in addition to the `RETRY_TEMP_STEP` bump the call also enables llama.cpp's DRY (Don't Repeat Yourself) sampler (`DrySampler(multiplier=0.8)` via `gate_common.RETRY_DRY_MULTIPLIER`) as a belt-and-suspenders catch for loops that survive 0.6+. DRY threads through `llm_client` (`generate_with_tool`/`stream_text` accept an optional `dry=DrySampler`, cascaded into the llamacpp convo exactly like `repeat_penalty`; re-exported from `llm_client`). It is **llamacpp-only** ‚Äî setting `dry` on a hosted model raises `UnsupportedFeature` in llmfacade ‚Äî so `gate_common._local_retry_dry(model)` gates it on the resolved backend (registry `provider == "llamacpp"`, mirroring `floor_for_local`) and returns `None` (never sent) for Anthropic/unknown models; the first attempt never sends DRY.

**Repetition-loop detection** is **delegated to llmfacade's `RepetitionGuard`** ‚Äî MTGAI no longer hand-rolls a detector (the old `theme_extractor._detect_tandem_repeat` + period bands were ported *into* llmfacade, so the guard's defaults reproduce them byte-for-byte: tail 4096, max_period 120, check_every 64). It scans assistant output text + tool-call args (NOT thinking blocks ‚Äî same scope as the detector it replaced). Configured **centrally** in `llm_client` via two module-level guards set on every **llamacpp** convo (never Anthropic ‚Äî cloud models don't loop): `_LLAMACPP_REP_GUARD` (tool `send` + `stream_text`; `escalate_dry=True`) and `_LLAMACPP_REP_GUARD_TEXT` (free-text `generate_text`; `on_exhausted="return_last"` to keep its "a partial reply is still useful, no raise" contract). Behaviour by call shape: `send` retries transparently (escalating repeat_penalty + DRY) then raises `RepetitionLoopError`; **`_generate_llamacpp` converts that to `OutputTruncatedError`** so every existing truncation-retry handler (`gate_common.generate_gate_tool`, reprint selection, slot grouper, mechanic-budget escalation) treats an unrecoverable loop exactly like a real truncation. `stream` aborts + raises mid-iteration; both stream consumers (`gate_common.stream_flag_batch`, `skeleton_relabel`) already catch it via their `except Exception` salvage-and-retry path. `theme_extractor` sets the guard on its own streaming convo and maps the raised `RepetitionLoopError` to its existing `repetition_abort` outcome. This is **detection** only ‚Äî the preventive temp-floor + the DRY/temp-on-truncation escalation above are unchanged. (llmfacade does not yet scan thinking/reasoning tokens ‚Äî a follow-up against the library.)

**LLM call logs**: llmfacade logging is on by default ‚Äî every `generate_with_tool` call writes a per-conversation JSONL **and HTML** transcript, named after its tool via `llm_client._convo_name()` (`select_best_mechanics-<uuid8>.html`, etc.) so logs are identifiable. Pass `generate_with_tool(log_dir=<dir>)` to route the transcript: the generators point it at their stage's `<asset>/<stage>/logs` (mechanics ‚Üí `mechanics/logs`, archetypes ‚Üí `archetypes/logs`, visual_refs ‚Üí `art-direction/logs`, card_gen ‚Üí `card_gen/logs`) so the transcript sits beside the stage artifacts. A Path writes flat into that dir (no `llmfacade<stamp>` session subdir, no auto-prune); omitting `log_dir` (default `None`‚Üí`True`) falls back to llmfacade's session dirs under `backend/logs/`. This llmfacade transcript replaced the hand-rolled per-generator logging (don't reinvent it). `card_generator` is the exception: it keeps its bespoke per-card/batch log for the post-generation **validation errors + applied fixes + cost** llmfacade can't capture, *and* routes the transcript to the same dir.
