## Project Structure
- Backend: `backend/mtgai/`. Tests: `backend/tests/` (mirrors source). Research: `research/`. Learnings: `learnings/`. Plans: `plans/`. Task tracking lives on Trello (see above) — `plans/TRACKER.md` is deprecated (history in git).
- All `output/` is gitignored. Pre-existing tracked files (ASD's `cards/`, `mechanics/`, `reports/`) stay tracked; new artifacts in `output/` won't be picked up.

## Development
- Python 3.12+ (uv-managed). Lint: `ruff check .` / `ruff format .` from `backend/`. Tests: `pytest` from `backend/`.

## Code Style
- Pydantic v2 for all data structures. StrEnum for enumerations.
- `X | Y` union syntax (not `Union[X, Y]`). `list[X]` (not `List[X]`).
- Line length 100. Imports sorted by Ruff (isort).

## Data Models
- Card schema is the single source of truth: `mtgai/models/card.py`. Field names match Scryfall (`oracle_text`, not `rules_text`).
- Card data: JSON files in `output/sets/<SET_CODE>/cards/`.
- Naming: `<collector_number>_<card_name_slug>.json`; art: `..._v<attempt>.png`; render: `....png`.

## Architecture overview

The codebase is a multi-stage pipeline that turns a setting prose document into a finished MTG set (cards + art + rendered images). Stages are orchestrated by `mtgai/pipeline/` and surfaced at `/pipeline` in the local FastAPI server (`python -m mtgai.review serve --open`). Each stage is independently runnable from the CLI.

Card statuses: `draft → validated → approved → art_generated → rendered → print_ready`. Pipeline is resumable; every generation attempt is tracked.

**Key modules** (look in the file for detail; CLI flags via `--help`):

- `mtgai/pipeline/` — end-to-end orchestrator: stage registry, FastAPI server, SSE event bus, wizard shell, configure page. State persisted to `<asset_folder>/pipeline-state.json` (the asset folder the user picked on Project Settings).
- `mtgai/pipeline/wizard.py` — wizard state resolver. Maps the on-disk `pipeline-state.json` + `theme.json` into the visible-tabs / latest-tab shape `wizard.html` consumes. URL routing lives in `pipeline/server.py` (`/pipeline/<tab_id>`). Project Settings (kickoff) tab payload comes from `GET /api/wizard/project`; live-apply edits go through `/api/wizard/project/{params,theme-input,breaks,models,preset/apply,preset/save}`; `POST /api/wizard/project/start` validates + spawns the theme extractor and tells the client to navigate to `/pipeline/theme`. The extraction worker, when launched via the kickoff path, also writes the assembled theme.json on `done` so the Theme tab finds populated content on next mount. **Building or modifying a wizard tab? Read `plans/wizard-tab-conventions.md` first** — it documents the patterns Project Settings + Theme establish (single Save & Continue footer button, required-field asterisks, form lock during AI gen, AI provenance badges, past-tab edit cascade, streamed-LLM textareas, etc.) so new tabs stay consistent.
- `mtgai/pipeline/theme_extractor.py` — theme content extraction. Surfaced as the Theme tab inside the wizard. Uploads PDF/text, runs multi-stage LLM extraction, then JSON pass for constraints + card suggestions.
- `mtgai/generation/llm_client.py` — all LLM transport routes through [llmfacade](https://github.com/Coamithra/LLMFacade) (Anthropic + llama.cpp). Single entry point `generate_with_tool()`; provider resolved per-call from the model registry.
- `mtgai/settings/` — model registry (`models.toml`) + the active project's `ModelSettings` (in-memory only; the .mtg file is the persistent store) + global default-preset (`output/settings/global.toml`). The settings shape carries `[llm_assignments]` / `[image_assignments]` / `[effort_overrides]` / `[break_points]` / `[set_params]` (set_name + set_size + mechanic_count) / `[theme_input]` (kind + filename + upload_id + char_count + uploaded_at) / `asset_folder`. All resolution goes through `get_active_settings()` (or `require_active_project().settings`); helpers no longer take `set_code`. The wizard's live-apply endpoints call `apply_settings(new)` to update the pointer; the user saves the .mtg via the browser's File System Access API to persist. Profiles save with `profile_only=True` so they exclude `set_params` + `theme_input` (they're per-project, not template-able). Cross-set defaults UI at `/settings`; per-project Project Settings UI is the wizard's first tab (`/pipeline/project`, endpoints under `/api/wizard/project/*`).
- `mtgai/validation/` — two-tier validators (AUTO auto-fixed; MANUAL becomes LLM retry prompts). Cards immutable; fixers return new instances.
- `mtgai/generation/reminder_injector.py` — reminder text is **never LLM-generated**, always injected programmatically from mechanic definitions after review.
- `mtgai/review/finalize.py` — post-review: inject reminder → validate → auto-fix → save.
- `mtgai/review/ai_review.py` — tiered LLM review (single-reviewer for C/U, full council for R/M + planeswalkers/sagas), iteration loop, resumable.
- `mtgai/analysis/` — balance checks (CMC curve, color balance, etc.) plus LLM-based interaction analysis flagging degenerate combos with named enabler.
- **Skeleton Generation** (`skeleton/generator.py` + `skeleton/knobs.py` + `generation/skeleton_knobs_tuner.py` + `generation/skeleton_relabel.py`, the `skeleton` stage between `archetypes` and `reprints`) — one stage, **three phases**, all under one AI-lock hold. (0) **Knob tuning** (`skeleton_knobs_tuner.tune_skeleton_knobs`): an LLM tunes the structural **knobs** (`SkeletonKnobs`) to the theme via one tool call — rarity weights, multicolor / colorless / creature fractions, non-creature type bias, planeswalker count, signposts-per-pair, and **cycles**. The LLM proposes; the schema clamps and `generate_skeleton` reconciles to the hard invariants, so an out-of-range value can never produce an illegal skeleton. Defaults reproduce the historical skeleton on the constraint-bearing dimensions (per-color balance, creature density, signpost coverage, rarity totals); `planeswalker_count` defaults to **0** so a failed/skipped tune leaves the failure-path skeleton unchanged. Failure → default knobs + `SkeletonResult.knobs_defaulted` (a tab notice, never a hard error). The knob schema (`skeleton/knobs.py`, `KNOB_SPECS` registry) is the single source of bounds driving AI-output validation, manual-input validation, and the wizard control bounds. **Cycles** (`Cycle`/`CycleSpan`) are balance-preserving structural reservations, always a multi-card family (one-per-color `mono5`, one-per-pair `pairs10`, allied/enemy fives, shard/wedge three-color fives — **no single-card span**; a one-off card is just a card, pinned via a theme `card_request`); they're carved out of the rarity budget first and stamp each member's `cycle_id` + `cycle_name` + `cycle_member` (the member's colour slot — `W` / `WG` / `WUB`). The name + member surface as a `· cycle:<name> (WG)` tag on `render_slot_string`, so the relabel sees the family *and each member's place in it* the same way it sees `signpost:<pair>`; the shared template is applied later at card-gen — this is how a set states a bold structural gesture (a 10-card guildgate land cycle, a mono-5 rare titan cycle) without breaking balance. (1) `generate_skeleton(config, knobs)` builds the deterministic, balanced **default** skeleton from the project's `set_params` (sized by `set_size`, *not* theme.json) and the tuned knobs. (2) `skeleton_relabel.relabel_skeleton()` runs the LLM **relabel**: each slot's one-line descriptor (`render_slot_string`, e.g. `"White · common · creature · CMC1 · vanilla"`) is rewritten to fit the set — Pass 1 relabels every descriptor (named mechanic instead of a bare tier, color/type/rarity swings, legendaries; count-invariant, reconciled by `slot_id`), Pass 2 places `theme.json` `card_requests` onto slots. **Pass 1 token-streams** via `llm_client.stream_text` (free text, not a tool call): each `--CARD <id>--` block is parsed the moment it closes and pushed to the Skeleton tab as a `skeleton_slot` SSE event (`skeleton_relabel_reset` per attempt, `skeleton_relabel_done` at the end), so cards fill in one at a time live — on both the engine auto-run and the manual Refresh, wired via `StageEmitter.event` / `event_bus.publish` + the `on_slot`/`on_reset` hooks on `relabel_skeleton`. The rewrite is stored per slot as `SkeletonSlot.tweaked_text`; the structured fields stay the default, so `reprints`/`lands` read them unchanged. Card generation reads `tweaked_text` as the slot's spec via `prompts.format_slot_specs` (else falls back to `render_slot_string`); the programmatic `group_slots_into_batches` batches by the (default) structured color **but pulls each cycle's members into their own batch** so the family is designed together with its shared `template` (threaded into the prompt as a CYCLE MEMBER instruction). **Land cycles** (cycle members carrying the deterministic `card_type == "land"` seed — e.g. a guildgate `pairs10`) are generated by **card-gen** like any other cycle, batched by `cycle_id` with their shared `template`. The `lands` stage (`land_generator.generate_lands`) is now just **basics + one investigated dual**: **2–4 alternate-art printings per basic type** (modern-set style — the basics call returns 4 distinct one-sentence art briefs + 1 flavor per type; a random 2–4 of the briefs become printings `L-01a`/`L-01b`/… each storing its brief in `design_notes` so the art stage renders a *distinct vista* per alternate, with flavor text on **at most one** printing per type) plus a single LLM **fixing investigation** that, seeing all three fixing sources (skeleton cycles, mana-fixing reprints from `reprint_selection.json`, and itself), adds **one rare from-scratch dual land** (`L-06`, `is_reprint=False`) only when a real multicolor gap remains or a strong gold theme warrants it. (Reprint placement is likewise offered *every* unfilled slot, avoiding cycle/signpost slots by prompt, not a structured type filter.) The Skeleton tab (`wizard_skeleton.js`) presents the stage's two halves as **two refreshable steps** (mirroring the Theme tab's section pattern): **Step 1 — Structural knobs** (bounded controls from `KNOB_SPECS`, provenance badges, pin, cycle list) whose **Refresh with AI** button re-tunes the knobs *and cascades into* a skeleton rebuild + relabel; **Step 2 — Skeleton**, which diffs each slot's default descriptor against its `tweaked_text` (word-level diff, editable) and whose **Refresh** button rebuilds the matrix from the *current* knob values *then* runs the LLM relabel. (Hand-edited knobs don't rebuild until a Refresh; the panel shows a "knob edits pending" hint meanwhile.) Endpoints `/api/wizard/skeleton/{state,refresh,save,knobs,knobs/tune}`, live stream via `W.onSkeletonStream`: `/skeleton/knobs` validates + deterministically rebuilds (no AI), `/skeleton/knobs/tune` re-runs phase 0 respecting pinned knobs (accepts the tab's current knobs as the tuner base), `/skeleton/refresh` runs the relabel. Both step refreshes orchestrate these client-side (rebuild → relabel, sequential awaits) so `local.slots` holds the rebuilt matrix before the relabel streams and each slot lands in the right row. The old single "Apply & rebuild" / "Tune with AI" buttons are gone — rebuild is now folded into the two Refresh cascades. Default skeleton is saved before the relabel, so a relabel failure leaves a usable un-themed skeleton. **Failure handling**: a partial relabel is *kept*, not discarded — unreached slots fall back to their default descriptor and the run is flagged `SkeletonResult.relabel_incomplete` (persisted on `skeleton.json`, surfaced as a tab warning that survives reload); only a zero-output run (every attempt errored) is a hard error. Model assignment `skeleton` (shared by phase 0 + relabel); transcripts → `<asset>/skeleton/logs`. The set-context prompt blocks (setting / mechanics / archetypes / constraints / requests) are shared between the tuner and the relabel via `generation/skeleton_prompt_blocks.py`. Prompts: `pipeline/prompts/skeleton_{knobs,relabel,assign}_{system,user}.txt`. Design: `plans/skeleton-generation.md` + `plans/skeleton-knobs.md`.
- **Reprint Selection** (`generation/reprint_selector.py` + `generation/reprint_knobs.py`, the `reprints` stage between `skeleton` and `lands`) — picks existing MTG cards from the curated `generation/reprint_pool.json` and places them on skeleton slots. **Two LLM passes, mirroring the skeleton relabel's select/assign split, and fully plain-text** (the reprint stage reads each slot's `tweaked_text`, not structured `color`/`rarity`/`type` — those are slot prose after the skeleton stage): (1) **Select** (`_select_from_pool`) — the set's theme + mechanics + archetypes + constraints (via the shared `skeleton_prompt_blocks` formatters) + the setting-agnostic pool → pick `count` staples that fit the set; (2) **Place** (`_place_reprints`) — the chosen cards + the plain-text slot list → assign each to an ordinary slot, retry/dedup like the relabel's Pass 2. **Avoiding "special" slots** (named characters, signposts, cycle members, build-arounds) is **prompt-engineered**, not structurally filtered — the descriptors carry `cycle:`/`signpost:` tags and rich prose the LLM is told to skip. **Count is per-rarity, soft** (`ReprintKnobs.{common,uncommon,rare,mythic}`): `None` = **auto** (the lean `REPRINT_RARITY_RATES` × the set's *estimated* rarity card count — `set_size × RARITY_WEIGHTS[r]/Σ`, since plain-text slots can't be counted by rarity), an int = **pinned** (exact). Auto rarities get a **proportional jitter** (`jitter_pct` of the auto total, realized on the largest auto rarity); `resolve_targets(knobs, set_size)` sums to a total. The breakdown is told to the select pass as soft guidance ("pick 1 rare, 3 uncommon, 4 common"); the pool cards carry their printed rarity so the model can honour it, but it's **not hard-capped per rarity** — a near miss is fine. Rates are tuned lean (below the 2.8% cross-set average; mythic lowest since its high research rate is nostalgia-driven, which a generated set doesn't do) → a ~277-card set auto-resolves to ~5 (3C/1U/1R/0M). Knobs persist in `<asset>/reprints/knobs.json`. `select_reprints(skeleton_path, knobs=None, count=None, temperature=0.0)`: an explicit `count` bypasses the knobs (flat total, scripts/tests); model is the active project's `reprints` assignment (**local Gemma by default**, Haiku under the recommended/all-haiku presets). Result → `<asset>/reprint_selection.json` (`per_rarity_targets` + `selections` = `{slot{slot_id,descriptor}, candidate, reason}`, where `descriptor` is the slot's text the reprint replaced); transcript → `<asset>/reprints/logs`. The Reprints tab (`wizard_reprints.js`) shows the picks (with "Replaces: <slot text>") + the per-rarity knob panel (auto/user badges, jitter, preview); endpoints `/api/wizard/reprints/{state,knobs,refresh}` — `/knobs` validates+persists (no AI), `/refresh` persists the knobs then re-runs under the AI lock at temperature 0.7 (manual re-roll surfaces alternates; the engine stage stays deterministic at temp 0). `clear_reprints` wipes the json + `reprints/` logs. **Note (pre-existing gap, tracked):** the selection is currently display-only — `convert_to_card` has no caller and nothing materializes the picks into card files; the materialization requirement (exact-copy reprint cards, flavor+art only, reprint-aware downstream) is a separate card.
- `mtgai/generation/skeleton_reviser.py` — post-balance reviser: LLM proposes slot changes from balance findings, regenerates affected cards. **Re-scoped to a lighter "did anything slip?" double-check** now that Skeleton Generation's relabel applies theme/constraints/requests up front (the theme-first prompt rework is tracked on card 6a13f98e). Its revision prompt's "Set Theme" section threads the active `theme.json`'s setting `constraints` + `card_requests` (normalized through `SetConfig`, covering the legacy `special_constraints` too) so revisions honor the setting.
- `mtgai/art/` — Flux.1-dev (via ComfyUI at `C:\Programming\ComfyUI`) for art generation, Haiku vision for art selection. PuLID-Flux for character face identity (humanoids only). `visual_reference_extractor.py` is the `visual_refs` pipeline stage (runs just before `art_prompts` — the art stages are its only consumers, so it sits with them rather than pre-skeleton): it reads `theme.json` prose and LLM-extracts `<asset>/art-direction/visual-references.json` — the per-project visual dictionary (`legendary_characters` / `creature_types` / `factions` / `landmarks` keyed dicts + `flux_term_replacements` + `visual_motifs`) that `visual_reference.py` and `character_portraits.py` consume.
- `mtgai/rendering/` — Pillow-based card compositor (M15 frames from Card Conjurer). Renders at 2010×2814, scales to 822×1122 (300 DPI).
- `mtgai/runtime/` — cross-cutting runtime (AI mutex, in-memory active-project pointer, broadcastable SSE for tab reattach). See "Cross-cutting" below.

## Cross-cutting

- **AI mutex (`mtgai/runtime/ai_lock.py`)**: app-wide, **one AI call at a time**. All AI-touching endpoints share it. Conflicting requests get 409 with a busy payload. New guarded endpoints wrap their body in `with ai_lock.hold("Action name") as acquired: if not acquired: return 409`. Long-running callers check `ai_lock.is_cancelled()` inside loops. **Strictly non-reentrant** (NOT an `RLock`): a thread already holding the lock that calls `hold()`/`try_acquire()` again gets busy back (`None`) and the nested body is silently skipped — acquire once at the outermost layer. `try_acquire()`/`hold()` return a monotonic `run_id` (`int` >= 1 on success, `None` when busy; truthy/`is not None` checks both work); `request_cancel(run_id=...)` only cancels if the active run matches that id, so a stale cancel can't kill a newer run that reused the lock. `request_cancel()` with no arg keeps the legacy "cancel whatever's running now" semantics (used by the UI cancel button + project-switch drain).
- **Tab state (`mtgai/runtime/runtime_state.py`, `extraction_run.py`)**: server is the source of truth for AI run state, pipeline state, and saved theme.json; browser holds UI ephemera in localStorage. SSE streams are broadcastable (tab-switching mid-run does not cancel; cancel is opt-in via the explicit button). New long-running AI runs should follow the same broadcast pattern. Aggregator: `GET /api/runtime/state`.
- **Active project**: which `.mtg` is open is the source of truth. The server holds a process-memory `ProjectState` pointer (`mtgai/runtime/active_project.py` — set_code + settings + .mtg path) — set by `/api/project/{open,materialize}`, cleared by `/api/project/new`, lost on restart so a fresh boot always greets the user with New / Open. Stage outputs land under the project's `asset_folder`; `mtgai/io/asset_paths.set_artifact_dir()` reads the active project and raises `NoAssetFolderError` (translated to 409 at endpoints) if none is open. Switching projects mid-AI-run requires `force=true` in the request body; without it the endpoint returns 409 + the AI lock's busy payload so the client can render a confirm modal. The cancel path calls `ai_lock.request_cancel()` then `active_project.await_lock_release()` before flipping the pointer.
- **Reminder text**: not validated, not LLM-generated. Stripped + injected by `reminder_injector` after review. Validators that touch oracle text must skip parenthesized text.
- **Progress UI**: the wizard's **global progress strip** (`#wiz-progress-strip` in `wizard.js`, sticky at top) is the one canonical progress surface. It's driven two ways: (1) SSE `phase` events on `/api/pipeline/events` → `handlePhaseEvent` — fed by `StageEmitter.phase()` for engine stages and by `phase_telemetry._emit_phase` for the only path with **live token-rate stats** (prompt-eval / generation tok/s), which today is `theme_extractor`'s streaming LLM calls; the generic `generate_with_tool` does **not** emit phase telemetry. (2) For synchronous, non-streaming AI tab actions that hold the AI lock but emit no telemetry (e.g. the Mechanics tab's "Re-pick with AI" + candidate refresh), call `MTGAIWizard.showBusy(label)` / `clearBusy()` — same strip, but an **indeterminate** animated bar (`wiz-progress-bar--indeterminate`) instead of a determinate fill. The AI mutex guarantees these two never overlap. Reusable by any tab; don't fake a determinate % when there's no real progress signal.

## Local LLMs
Local models run through llmfacade's `llamacpp` provider in managed mode (lazy-spawns `llama-swap`). Registry: `backend/mtgai/settings/models.toml`. For setup, gotchas, and adding a model, see `learnings/local-llm-setup.md`. For benchmarks: `learnings/llamacpp-tc2-benchmark.md`. For the Gemma repetition-loop story: `learnings/gemma-repetition-loops.md`.

**LLM call logs**: llmfacade logging is on by default — every `generate_with_tool` call writes a per-conversation JSONL **and HTML** transcript, named after its tool via `llm_client._convo_name()` (`select_best_mechanics-<uuid8>.html`, etc.) so logs are identifiable. Pass `generate_with_tool(log_dir=<dir>)` to route the transcript: the generators point it at their stage's `<asset>/<stage>/logs` (mechanics → `mechanics/logs`, archetypes → `archetypes/logs`, visual_refs → `art-direction/logs`, card_gen → `card_gen/logs` via `card_generator._card_gen_log_dir()`) so the transcript sits beside the stage artifacts. A Path writes flat into that dir (no `llmfacade<stamp>` session subdir, no auto-prune); omitting `log_dir` (default `None`→`True`) falls back to llmfacade's session dirs under `backend/logs/`. This llmfacade transcript replaced the hand-rolled `_save_generation_log` in mechanic/archetype/visual_ref generators (don't reinvent it). `card_generator` is the exception: it keeps its bespoke per-card/batch log for the post-generation **validation errors + applied fixes + cost** llmfacade can't capture, *and* routes the transcript to the same dir.
