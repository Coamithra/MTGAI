# Phase 2A-2 Learnings: Art Prompt Pipeline

## What We Built

- `backend/mtgai/art/prompt_builder.py` — Flux-optimized prompt generation pipeline
- `backend/mtgai/art/visual_reference.py` — JSON-driven visual reference loader
- `output/sets/ASD/art-direction/visual-references.json` — canonical visual descriptions
- `output/sets/ASD/art-direction/prompt-templates.md` — template documentation
- `output/sets/ASD/art-direction/style-guide.md` — full art style guide

## Key Learnings

### 1. Flux Prompts Must Be Short and Front-Loaded

Our first attempt produced 150-200 word prompts with 6 stacked layers (style persona + universal prefix + LLM description + rarity modifier + mechanic visual + universal suffix). This is wrong for Flux.

**Flux best practices (from BFL docs):**
- Front-load the subject — Flux pays most attention to what comes first
- 30-80 words sweet spot (we target 40-60)
- Natural language, not keyword lists
- No negative phrasing ("no text, no border") — describe what IS there
- Structure: Subject + Style (not Style + Subject)

**Final architecture**: `[Haiku visual description]. [One-line style directive].` ~60-80 words total.

Output tokens dropped 48% (13,341 → 6,985). Cost dropped $0.13 → $0.10.

### 2. Image Gen Models Don't Know Your Setting

This is the biggest lesson. Flux has no idea what a "Moktar" or "Morlock" is. Character names like "Koyl Yrenum" are meaningless noise. The pipeline needs two layers of defense:

1. **Haiku gets visual references** — setting-specific creature/faction/character descriptions injected into the LLM context so it can translate "Moktar Warrior" → "seven-foot muscular humanoid with tawny fur and lion-like maned head"
2. **Post-processing sanitizer** — catches any setting-specific terms Haiku lets through and replaces them with plain English (safety net)

Both the references and replacements are loaded from `visual-references.json` (data-driven, not hardcoded).

### 3. theme.json Must Include Visual Descriptions

Our `theme.json` defines creature types and factions but never describes what they *look like*. The art pipeline had to pull physical descriptions from the source material PDF and card flavor text.

**For automation**: Phase 1A (set design) should output visual descriptions for every named entity — races, factions, characters, landmarks, legendary items. This becomes the single source of truth that the art pipeline reads.

### 4. Character Consistency Requires Reference Images

Text descriptions alone can't ensure the same character looks consistent across multiple cards. The plan:
- Generate neutral reference portraits per legendary character (Flux dev text-to-image)
- Human picks best version
- Use Flux Kontext Dev + PuLID during card art generation
- PuLID extracts only face identity, ignoring reference style — text prompt controls all styling

This is critical because IP-Adapter has "style leakage" (reference image style bleeds into output), while PuLID mathematically separates identity from style.

### 5. The LLM Is the Best Translator

Haiku at $0.10/set translates MTG jargon → visual English perfectly. It knows what "vigilance" means as a visual concept without being told. The key instructions:
- Don't use character names (describe appearance)
- Don't use made-up race names (describe the creature)
- Don't use game terminology
- Target 40-60 words
- Temperature 0.6

### 6. Data-Driven > Hardcoded

First implementation hardcoded all visual references in Python. Moved to JSON (`visual-references.json`) so:
- It's set-agnostic (works for any set, not just ASD)
- It can be generated by the LLM during set design
- Non-programmers can edit it
- The Flux term replacement map lives in the same file

### 7. Consult Existing Research First

We had `research/art-prompt-comparison.md` (Haiku selected, temp 0.6 recommended) and `research/proof-of-concept/image-gen-test-prompts.md` (scoring rubric, test prompts) that were directly relevant but weren't consulted until mid-way through. Always read all learnings and research files before starting a new phase.

## Cost Summary

- Style guide + prompt templates: manual (no API cost)
- Prompt generation v1: $0.13 (66 cards, 13,341 output tokens)
- Prompt generation v2 (Flux-optimized): $0.10 (66 cards, 6,985 output tokens)
- Total Phase 2A API spend: ~$0.23

## Input/Output Spec for Automated Pipeline

**Inputs:**
- `theme.json` with `visual_descriptions` section (races, factions, characters, landmarks)
- `mechanics/approved.json` (mechanic flavor)
- Card JSONs with name, type_line, oracle_text, flavor_text, design_notes, colors, rarity
- Image gen model choice (Flux dev vs DALL-E vs Midjourney — affects prompt style)

**Outputs:**
- `art-direction/style-guide.md` (narrative)
- `art-direction/visual-references.json` (creature types, factions, landmarks, characters, flux replacements)
- `art-direction/prompt-templates.md` (documentation)
- `art_prompt` field populated on every card JSON
- `art-direction/prompt-logs/*.json` (per-card logs)
- `art-direction/character-refs/` (reference portraits — Phase 2A-5)
